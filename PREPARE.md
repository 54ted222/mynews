# PREPARE · 雜誌主題管線

本檔為「下期雜誌主題」的儲備腳本。執行時讀取並維護專案根目錄的
`TOPIC.md`，把主題從粗到精，沿著**建議 → 預備 → 下期**三層漏斗往
下推。每一層都有容量上限，不足時從上一層挑一個補進來並細化。

與其他腳本的關係：

- **PREPARE.md（本檔）**：只管 `TOPIC.md`，養主題池、挑下期題目
- **MAGAZINES.md**：拿到「下期主題」後開始策劃、寫稿、發刊
- **ROUTINES.md**：每日報紙，與本檔無直接耦合

也就是說：**本檔的產出是 `TOPIC.md` 裡那一條下期主題**，它就是下一次
跑 MAGAZINES.md 時的題目起點。

---

## 讀者輪廓（從 `TOPIC.md` 的「興趣 interest」讀取）

> ⚠️ **interest 區塊全程唯讀，不要寫入、不要重寫、不要擅自補內容**。

執行前先把 `TOPIC.md` 的 `## 興趣 interest` 整段讀進 context，當作
本次補給的**讀者輪廓錨點**——所有新增或細化的主題，都必須能回答
「這位讀者會想看嗎？」。若此區塊為空，主 agent 必須**停下來告知
使用者**：「interest 區塊尚未填寫，請先寫入讀者輪廓再執行 PREPARE」，
**不要**憑空想像讀者。

## `TOPIC.md` 結構規格（嚴格遵守）

整份 `TOPIC.md` 固定四個 H2 區塊，順序不變、標題不變：

```md
# TOPICS

## 興趣 interest

<讀者輪廓，自由格式，PREPARE 不動>

## 建議主題 suggested topic

<20 個項目，每項約 50 字>

## 預備主題 prepared topic

<10 個項目，每項約 200 字>

## 下期主題 next topic

<1 個項目，約 500 字>
```

### 建議主題（suggested topic）— 20 項

- **單行格式**，約 50 字（含關鍵字在內）：

  ```md
  - **<主題標題>** — 關鍵字：<kw1>, <kw2>, <kw3>, <kw4>
  ```

- 標題用粗體；關鍵字 3–5 組，用逗號分隔，繁中或英文皆可
- 一項一行，維持 bullet list 的單層縮排
- 順序不具意義，但建議把最近新補的放在最上面，方便使用者看到變動

### 預備主題（prepared topic）— 10 項

- 每項用 H3 + 兩段組成，總字數約 200 字：

  ```md
  ### <主題標題>

  關鍵字：<kw1>, <kw2>, <kw3>, <kw4>, <kw5>

  描述：<150 字左右的短段落，說明這個主題為什麼值得寫、讀者會在意
  哪些切面、目前產業／社群的溫度、可能切成的 2–3 條主線。>
  ```

- 關鍵字 5–7 組；比 suggested 更具體（例如不是「AI 定價」而是
  「Cursor 2026 pricing backlash」）
- 描述段不要放連結、不要放 bullet，就是連貫的短文

### 下期主題（next topic）— 1 項

- 固定 H3 + 四段，總字數約 500 字：

  ```md
  ### <主題標題>

  關鍵字：<kw1>, <kw2>, ..., <kw8>

  短文：<300 字左右，說明本期雜誌的定位、動機、主軸、預計切出的
  3–5 條主線的走向，以及為什麼「現在」要做這題。>

  參考來源：
  1. [<標題>](<URL>) — <媒體/作者>, <YYYY-MM-DD>
  2. [<標題>](<URL>) — <媒體/作者>, <YYYY-MM-DD>
  3. ...
  ```

- 關鍵字 6–10 組，必須是實際 WebSearch 查得到新結果的具體 query
- 短文要已經能讓 MAGAZINES.md 階段一直接接手；換句話說，它就是下
  一期雜誌 `_index.md` 的 intro 草稿
- 參考來源 3–6 條，以一手／一手討論為主（見 MAGAZINES.md「資料來
  源優先順序」）

## 執行流程（由下往上補，順序不可換）

本腳本的核心邏輯：**下層缺，就從上層取一個、細化、搬下去**。因此
執行順序是先補下層、再回補上層；每一層都做一次「盤點 → 補給 →
寫回」。

執行前：

1. 用 `Read` 讀入 `TOPIC.md` 全文
2. 把四個區塊切開解析，確認結構沒被外力破壞（缺 H2、順序錯亂都要先修）
3. 記下目前各區塊的項目數 `n_suggested`、`n_prepared`、`n_next`
4. **早退檢查**：若以下條件**全部成立**，直接回報「主題池已滿，無需補給」
   並結束，**不要**進入第 1–3 步、不要覆寫 `TOPIC.md`：
   - 數量齊全：`n_suggested == 20`、`n_prepared == 10`、`n_next == 1`
   - **沒有只有標題的殘件**——每一項都具備對應格式的完整欄位：
     - 建議主題：粗體標題 **+** 關鍵字那一段都在
     - 預備主題：H3 標題 **+** 關鍵字行 **+** 150 字左右描述段
     - 下期主題：H3 標題 **+** 關鍵字行 **+** 300 字左右短文 **+** 3–6 條參考來源
   - 下期主題不是 `<完整的主題名稱> : ...` 這類 placeholder

   只要任一條件不成立（數量不足，或有「只剩標題、沒關鍵字／沒描述／沒
   來源」的殘件），視為需要補給，繼續第 1–3 步。

### 第 1 步：補「下期主題」至 1 項

- **若 `n_next >= 1` 且該項目已符合 500 字 + 標題／關鍵字／短文／來
  源四段齊全**：跳過本步。
- **若只是 placeholder（例如 `<完整的主題名稱> : ...`）或格式不齊**：
  視為空，往下補。

補給規則：

1. 從「預備主題」中挑 **1 個最適合當下期**的主題（判準：新聞熱度、
   讀者輪廓契合、與近期已發刊雜誌不重疊、資料可查性）
2. 對該主題 **WebSearch 3–6 次**、對關鍵來源 **WebFetch 讀全文**；挑
   到的主題若「資訊太薄、太舊、全二手」，回到第 1 步換下一個候選
3. 把它改寫成**下期主題格式**（標題、關鍵字、300 字短文、3–6 條來源）
4. 同步**從「預備主題」區塊移除該項**（避免重複）
5. 寫入「下期主題」區塊，取代原 placeholder

> 若預備主題區塊一項都沒有，先往下走第 2 步把預備補出來，再回
> 來做第 1 步。

### 第 2 步：補「預備主題」至 10 項

- 盤點目前 `n_prepared`；若 `n_prepared >= 10` 跳過。
- 否則需要補 `10 - n_prepared` 個。

補給規則：

1. 從「建議主題」中挑出最有潛力的幾個（判準：與讀者輪廓契合、有
   近期動態可寫、與現有預備主題不重疊）
2. 對每個候選 **WebSearch 2–4 次**，確認有料可寫
3. 把每一個擴寫成**預備主題格式**（H3 標題、5–7 組關鍵字、150 字
   描述），寫入「預備主題」區塊
4. 同步**從「建議主題」區塊移除已搬上來的項目**

> 若建議主題不夠挑，先往下走第 3 步把建議補出來，再回來做第 2 步。

### 第 3 步：補「建議主題」至 20 項

- 盤點目前 `n_suggested`；若 `n_suggested >= 20` 跳過。
- 否則需要補 `20 - n_suggested` 個。

補給規則：

1. 以「興趣 interest」為錨，把讀者輪廓的關鍵字展開成搜尋面向
2. **WebSearch 3–6 次**掃近期的產業動態、討論熱點、工具更新、觀點
   文章；同時參考現有的預備／下期主題，沿著相關主題外擴（例如現
   有預備是「RAG 實作細節」，可以外擴「向量庫選型」「Embedding
   成本下探」「RAG 評測方法」）
3. 產出新的建議項目，每項一行、約 50 字，含標題與 3–5 個關鍵字
4. **去重**：若與現有 suggested／prepared／next 的主題重疊，跳過
5. 寫入「建議主題」區塊；新增的放清單**最上面**，方便使用者一眼
   看到本次新增了什麼

## 寫回規則

- **整份 `TOPIC.md` 用 `Write` 覆寫**（本檔短，整檔寫最簡單安全），
  不要用多段 `Edit` 拆開改
- **interest 區塊原文原樣保留**（含裡面的空白行、格式），不 trim、
  不重排
- 四個 H2 的標題與順序不變；每個 H2 下空一行再接內容
- 區塊之間用單一空白行分隔；不要在檔尾留多餘空白行
- 若本次執行沒改動到某個區塊，仍按原樣寫回，確保全檔一致

## 研究要求（嚴禁憑空想像）

與 MAGAZINES.md 同調：

- 所有**新增或細化**的主題，其具體數字、公司名、產品名、事件，都
  必須經過 WebSearch / WebFetch 驗證
- 訓練資料有截止日，本站今天（執行日）在跑的題目常是近期動態——
  **不要**靠記憶補主題
- 下期主題的「參考來源」必須是**實際打開讀過**的連結，不是憑印象
  拼湊
- 若某個候選主題查下來資訊太薄或全是二手轉載，**換下一個候選**，
  不要硬上
- 來源優先順序見 MAGAZINES.md 的「資料來源優先順序」章節

## 交付步驟

1. 讀入 `TOPIC.md`，解析四區塊、盤點數量
2. 依第 1 / 2 / 3 步順序補給（若下層依賴上層先補的，先切到上層）
3. 用 `Write` 覆寫 `TOPIC.md`
4. 回報 200 字內：
   - 各區塊補了幾個、移了幾個
   - 下期主題是哪一個（標題）、從哪裡搬上來的
   - 有沒有遇到「資訊不足必須換題」的情形
5. **不要**執行 git、**不要**跑 `npm`、**不要**動 `public/` 或 `src/`

> PREPARE 只管 `TOPIC.md` 這一個檔案；交付完停手。是否要 git
> commit 由使用者決定（通常會等下期雜誌真的寫完一起推）。

## 驗收檢查（產出前自審）

- [ ] 本次執行前實際讀了最新版 `TOPIC.md`，不是從 context 沿用
- [ ] 「興趣 interest」區塊**一字未動**（或仍為空則已告知使用者暫停）
- [ ] 「建議主題」恰好 20 項；每項一行、含粗體標題與關鍵字
- [ ] 「預備主題」恰好 10 項；每項 H3 + 關鍵字 + 150 字描述
- [ ] 「下期主題」恰好 1 項；標題 + 關鍵字 + 300 字短文 + 3–6 條來源
- [ ] 「建議」被挑走的項目已從原區塊移除（不會同時出現在建議與預備）
- [ ] 「預備」被挑到下期的項目已從原區塊移除
- [ ] 下期主題的來源連結是實際讀過的一手 / 一手討論，不是憑印象
- [ ] 所有新增項目與現有項目**沒有語意重複**
- [ ] `TOPIC.md` 四個 H2 標題與順序維持不變
- [ ] 沒有動到 `src/`、`public/`、`package.json`、git、`npm`

## 反模式（請避免）

- ❌ 擅自改寫 / 補充 interest 區塊
- ❌ 建議主題膨脹到 20 項以上、或項目寫到 100 字以上（應是 50 字的快照）
- ❌ 預備主題的描述寫成 bullet 清單或跑到 400 字（那是下期主題的規格）
- ❌ 下期主題沒有來源段，或來源是憑印象寫出的 URL
- ❌ 從建議搬到預備、或從預備搬到下期後，**忘了把原項目刪掉**，導致重複
- ❌ 靠記憶補主題池，沒跑任何 WebSearch
- ❌ 順手 git commit / push（本腳本不處理 git）
- ❌ 順手動 `public/magazines/`（那是 MAGAZINES.md 的地盤）
