前面三集我們談了護欄要擺在哪一層、還有動手那一刻怎麼約束。今天這一集換個視角，從採購的角度來看：每一層，市面上到底有什麼東西可以直接拿來用，它們的延遲、授權、強項跟坑各在哪裡。假設你已經懂分層了，這集只回答「用什麼做、怎麼權衡」。

結論我先講在前面：護欄工具鏈不是「選一個框架」的單選題，而是把不同性質的檢查，擺到不同延遲預算的格子裡。便宜的正則跟分類器擋在最前面，貴的大模型判官只在必要時才升級上去。沒有哪個工具能單獨扛下全部，這點要先放在心裡。

先講最根本的兩條路線之爭：是用規則去描述什麼不能過，還是用一個模型去判斷這次該不該過。規則式這條路，涵蓋正則、關鍵字、格式驗證、個資偵測，還有訓練來做單一決策的小型分類器。它的好處是延遲低、成本近乎零、而且輸出可解釋——你知道它為什麼擋。正則驗證以微秒計，神經網路分類器大概落在數十到數百毫秒。壞處就一個字：脆。攻擊者換個說法就能繞過，碰到沒見過的新手法幾乎無能為力。

另一條路是大模型判官，就是用另一個語言模型去評估主模型的輸入或輸出——看它有沒有所本、相不相關、語氣對不對、有沒有被注入。它能讀懂語意、能抓到規則寫不出來的新型攻擊，但代價很直接：每一次評估，就是多跑一輪完整的生成，通常會增加兩百到一千毫秒，等於把一次生成放上了使用者等待的關鍵路徑。而且它也會誤判，判斷理由還不如規則透明。所以二零二六年的主流做法不是二選一，而是分層升級：先跑便宜的快檢，命中可疑訊號才升級到重檢。這裡有個隱形成本要提醒你，就是 token。一個四層、每層都呼叫大模型的護欄堆疊，對一筆一千 token 的查詢，常常會額外吃掉兩千到四千個 token，這在帳單跟延遲上都會反映出來。

接下來我幫你盤點幾個主要的工具，重點是搞清楚誰負責哪一層。你可以把它們分成兩類：一類是編排框架，定義你怎麼把各層串起來；另一類是分類器模型，是其中某一個關卡真正執行判斷的那顆腦。

第一個，NeMo Guardrails，是 NVIDIA 開源的，用的是寬鬆的 Apache 2.0 授權。它把護欄分成五種類型：輸入、對話、檢索、執行、輸出。它最大的差異化在「對話」這種——它用一個叫 Colang 的專用語言，把整段對話流程建模出來。多數工具只能過濾單次的輸入輸出，NeMo 能管「對話走到哪一步、此刻該不該觸發某個護欄」。代價是 Colang 是一個真正的領域特定語言，有自己的概念，團隊普遍低估了它的上手成本，新版本刻意往 Python 語法靠攏就是想壓低這條學習曲線。延遲方面，模式比對那種 rail 可以做到一百毫秒以內，在 GPU 上甚至更快，但要注意那個漂亮數字是假設你有 GPU，跑 CPU 要重新估算。

第二個，Guardrails AI，也是開源、Apache 授權。它是一個 Python 跟 JS 的框架，核心是它的 Hub——上面有七十多個現成的驗證器，涵蓋個資、越獄、事實性、格式、程式碼漏洞、品牌風險，你組一組就能攔住模型的輸入跟輸出兩端。它的強項是即插即用、生態廣，輸出驗證的規模比 NeMo 大。它甚至還出過一個跨類別比較二十幾個護欄表現跟延遲的 benchmark，是少數有第一方數據的框架。弱項是它比較偏「輸入輸出過濾」，不像 NeMo 那樣能建模整段對話狀態。

第三個跟第四個是分類器模型，不是框架。一個是 Meta 的 Llama Guard 4，120 億參數，對齊一套標準的危害分類，支援多語跟圖文混合輸入。但要注意它的授權不是純開源，月活超過七億的大產品要另外向 Meta 申請。它的定位就是「被某個關卡呼叫的那顆安全分類腦」。另一個是 OpenGuardrails，這個是全開源的，主打用單一一個模型同時做內容安全偵測跟操弄防禦，還會輸出一個信心分數，讓管理者依自己的風險容忍度去調門檻。它的模型比較小、論文回報延遲在三百毫秒以內，定位是低延遲、可以地端部署，適合那種想要私有部署、又想用單一模型涵蓋多類風險的團隊。

那到底什麼場景選什麼？我給你幾個簡單的判斷。如果你是對話型 agent、需要管「對話走到哪一步才觸發護欄」，那 NeMo 的對話 rail 值得你吃下 Colang 的學習成本；但如果你只是想進來擋一下、出去驗一下，那 Colang 多半是過度投資。如果你想快速補上輸出驗證、團隊又以 Python 為主，那 Guardrails AI 的 Hub 是摩擦最低的起點，記得把那種大模型判官類的驗證器留給非同步監控，別塞進關鍵路徑。如果你要的就是一顆能擋有害內容又能擋注入的分類器，那就在 Llama Guard 4 跟 OpenGuardrails 之間二選一，看你在意純開源、要地端、想調門檻，還是已經在 Llama 生態、需要多模態。

最後講一個很多人會踩的坑，就是開源跟商用的落差。本集談的開源工具，省的是授權費，付出的是維運——追蹤的串接、離線回歸測試、儀表板都得自己寫，延遲也要自己拿硬體去頂，剛剛說 NeMo 那個八十毫秒是假設你有 GPU，跑 CPU 上就會變樣。商用的 runtime 把這些都封裝起來，代價是綁定跟費用。沒做過容量跟延遲預算就直接上開源框架，常常會在正式環境踩到硬體跟維運的隱形成本。

所以這一集的重點是：工具鏈是按延遲預算分格子，不是選一個贏家。但不管你選了什麼，這些護欄到底擋不擋得住，光靠自己宣稱沒有用，要靠紅隊跟評測去證明。那「怎麼證明護欄真的有效」，就是我們最後一集要面對的、也是最難的問題。
