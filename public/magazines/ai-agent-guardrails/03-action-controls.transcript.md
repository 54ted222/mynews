前兩集我們談了在防什麼、還有護欄該擺在架構的哪一層。今天這一集，我們把鏡頭推到最近、也最危險的距離——就是 agent 已經決定要呼叫某個工具、要送出某個動作，就在它動手的那一毫秒。這一層要是沒守住，前面的輸入過濾、推理監控全都白搭，因為真正會寄出信、刪掉資料、轉出款項的，就是這一刻的工具呼叫。

行動層護欄不是一個單一開關，而是一條控制鏈。有一份白皮書把它整理成四個要素，我覺得很好記：權限，這是事前預防；核可，這是人類介入；稽核軌跡，這是事後可追；還有緊急開關，這是事中應變。我們今天就沿這四個要素走一遍。

先講權限，也就是最小權限原則。最便宜、也最常見的災難，就是團隊圖省事，直接開一個全域的服務帳號讓 agent 用。後果是什麼？任何能跟這個 agent 對話的人，都間接握有那個帳號的全部權限。有調查指出，對 AI 系統過度授權的企業，遭遇的安全事件數量是嚴格控管者的四點五倍。這不是理論，是已經發生的代價。

正確的起點是把權限按元件拆開、按需要給。負責規劃的那塊盡量設成唯讀，它只要讀資料來想方案，不需要寫入能力；真正執行的部分，只拿到當前任務需要的那幾個工具；至於那些能改系統、能拿到密鑰的高敏感權限，目標是「零常駐權限」——平常一點都不給，任務開始才授予、任務結束就收回。把可以被利用的時間窗，從無限期壓到接近零。

不過比「給什麼權限」更關鍵的，是「用誰的身分」。這裡有兩條路，差距很大。我用講的對比一下。第一條路叫委派身分，也就是 on-behalf-of token。它的權杖裡同時記了兩個身分：被授權的那個使用者、跟實際執行的那個 agent。好處是 agent 永遠不可能超過使用者本人的存取範圍——Alice 只看得到她自己的行事曆，那代她行動的 agent 也就只看得到她的。而且範圍明確、可以即時撤銷、可以設短效期，稽核紀錄也很誠實，誰委派、哪個 agent 動的手，都在。第二條路就是剛剛講的全域服務帳號，單一共用身分、看不出背後是誰、全域權限、所有請求都用同一個全能身分跑，一撤就全停，稽核也只看得到「服務帳號做了某件事」。這兩條一比，高下立判。委派身分能擋掉一整類問題：避免 agent 累積了使用者的全部權限,但它的任務其實只需要其中一小片。

第二個要素是動手前的三道閘。最小權限決定了 agent 手上有哪些工具可用，但光這樣不夠，因為模型可能選對了工具卻用錯方式，或是該用 A 工具卻選了 B。所以在模型產出工具呼叫之後、真正執行之前，要再過三道彼此獨立的檢查。第一道是工具白名單，問的是「這個工具，此時此地能不能用」，擋掉呼叫了不在授權清單裡的工具，比方說這個任務只允許讀報告跟送摘要，就不允許刪紀錄。第二道是 schema 驗證，問的是「參數的型別、必填、格式對不對」，比方說收款帳號必須符合標準格式、金額必須是正整數。第三道是不變量檢查，問的是「有沒有踩到範圍、門檻、業務規則」，比方說單筆轉帳上限一萬、檔案路徑必須在某個目錄底下、禁止碰到含有 secret 字樣的檔案。這三道的精神是：它們完全不信任模型的輸出。白名單擋「不該碰的工具」、schema 擋「結構壞掉的呼叫」、不變量擋「結構正確、但語意危險的呼叫」，三個分工、缺一不可。

第三個要素是人類核可。有些動作再多自動檢查都不該全自動放行——金融交易、對外溝通、刪除資料、特權變更。這時候要把人放回迴圈裡。常見做法是按風險把動作分級：純讀取的低風險可以自動放行；寫入、API 呼叫是中風險；金融、刪除、對外通訊是高風險；不可逆、資安敏感的像是刪資料庫、轉帳，是最高風險，必須人類點頭。這裡有兩個容易被忽略的細節。第一，決策跟執行要分離，agent 只能提案，由一個獨立的服務在執行前再驗一次，這樣即使 agent 被注入騙過，它也沒辦法自己批准自己。第二，核可要綁定「這一筆」動作，紀錄裡要寫清楚是誰、哪個工具、目標資源、正規化後的參數、時間戳跟到期時間，不然一個「同意送信」的核可，可能被偷換成送另一封信。非同步的做法業界多用一個叫 CIBA 的標準，由像 Auth0 那樣的產品實作：agent 發起請求，授權伺服器透過推播通知你，你在自己的裝置上按確認，agent 收到結果才執行。

但我要強調這一集最重要的取捨：人類核可不是越多越好。塞太多核可，就會得到所謂的「核可疲勞」——使用者被一堆瑣碎核可轟炸，最後不看內容就一路按同意，監督就名存實亡了。這跟資安維運中心二十年來的「警報疲勞」是同一個病。對策有三個：一是按風險分級，只在罕見又後果重的動作上攔人；二是設計批次審核，讓 agent 跑完一整個工作單元，人類審一份彙整好的變更清單，而不是被二十個彈窗逐一打斷；三是接受「核可負擔」跟「風險下降」本來就是要量化權衡的，沒有免費的安全。

最後是緊急開關，kill switch。它不是「砍掉 agent 進程」這麼簡單，而是一整套圍堵模式：撤銷身分、停掉工具存取、凍結編排、把系統推回已知的安全狀態，比方強制變成唯讀。這裡有一條鐵律：緊急開關必須跑在 agent 的推理路徑之外，不能依賴模型輸出或 agent 邏輯，要由編排層或基礎設施來強制執行——否則一個被攻陷的 agent，可能連自己的開關都關掉。而且這已經不只是工程選擇了：歐盟的 AI 法案第十四條就要求高風險 AI 系統必須讓人能監督、介入、並停止它的運作，主要條文二零二六年八月生效，等於把「人能喊停」從最佳實踐推成了法律義務。

所以把這四件事串起來——最小權限收窄能做什麼、三道閘驗證怎麼做、人類核可擋住最重的決定、緊急開關兜住最壞的情況——agent 才從一個「動起來就收不回」的黑盒，變成一個可停止、可追蹤、可修正的系統。那道理都懂了，市面上到底有哪些工具能幫你把這些做出來？這就是下一集工具鏈要回答的問題了。
