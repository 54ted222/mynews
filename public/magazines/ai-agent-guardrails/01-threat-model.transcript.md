這一集想聊一個很基本、但常常被跳過的問題：當我們說要幫 AI agent 裝護欄，我們到底在防什麼？因為你要是搞不清楚威脅長什麼樣，後面裝再多東西都是憑感覺。

先講最核心的一件事。聊天機器人跟 agent 最大的差別，不是誰比較聰明，而是後果的型態完全不同。聊天機器人的錯誤是可讀的、可逆的、需要人轉手才會生效——它說錯一句話，你看到、你不照做，就沒事了。可是 agent 的錯誤是自動的、可能不可逆的、不經人手就生效——它決定要寄一封信，信就出去了；它決定要刪一個分支，分支就沒了。原本卡在決策跟執行之間的那個人類，被抽掉了。這就是我們整期一直在講的「爆炸半徑」放大。每多接一個工具、每放寬一級權限、每省掉一道人類核可，攻擊面就大一圈，爆炸半徑就遠一截。

那威脅的入口在哪？答案是提示注入。這個詞是資安研究者 Simon Willison 他們在二零二二年提出的，刻意拿來對照大家熟悉的 SQL 注入，因為底層是同一個病：把可信的內容跟不可信的內容混在同一個輸入裡，系統分不清哪一段是資料、哪一段是指令。對語言模型來說這更難治，因為所有進到它腦袋裡的字，地位都是平等的，沒有一個跳脫字元可以把使用者的資料跟系統的指令隔開。

提示注入有兩種，要分開看。第一種叫直接注入，就是使用者自己在輸入框打惡意指令，這種情況下使用者本人就是攻擊者，多半是想越獄，危害大致侷限在他自己。第二種比較恐怖，叫間接注入：惡意指令藏在 agent 會去讀的外部內容裡，可能是一個網頁、一封郵件、一張客服工單、甚至另一個工具回傳的東西。這種情況下使用者是受害者、不是攻擊者。agent 自動把這些外部文字吞進去，攻擊者根本不用碰到你的鍵盤。這就是經典的混淆代理問題——攻擊者自己沒有權限，但他能讓那個有權限的 agent 替他動手。

這裡我必須很誠實地講一個結論：截至二零二六年年中，提示注入沒有被徹底解決，短期內也不會。OpenAI、Anthropic、Google DeepMind 這三大廠的研究共識都是，在現有的語言模型架構下，任何你用提示寫的防禦，本身都可能被另一個提示蓋掉。Google DeepMind 提過一個叫 CaMeL 的設計，用兩個分工的模型加上能力追蹤，把防禦從靠模型自律推向比較確定性的程式控制，這算是很認真的嘗試了，可是在標準測試上也只擋下大概六成七的注入。連設計上最嚴謹的方案都不敢說清零。所以這就是我們所有護欄工程的前提：我們不是在追求擋住每一個注入，而是在假設注入一定會發生的前提下，去限制它能造成的後果。

講了這麼多抽象的，不如看一個真實跑通的案例，就是 EchoLeak。這是二零二五年六月揭露的，目標是 Microsoft 365 Copilot，編號 CVE-2025-32711，嚴重度給到九點三，非常高。它最可怕的地方是「零點擊」——受害者什麼都不用做。攻擊者只要寄一封郵件進公司，剩下的 Copilot 自己會完成。它其實串了四個各自看起來都不致命的繞過：先用措辭規避掉微軟偵測注入的分類器、再用一種特殊的連結寫法躲過連結遮蔽、然後利用客戶端會自動預抓圖片這件事，讓一張圖片就觸發一個對外的網路請求把資料偷送出去、最後還借道一個被信任的 Teams 預覽介面當跳板繞過內容安全政策。研究者甚至還用了一招，把惡意內容塞滿檢索空間，確保不管你問什麼問題，那封壞郵件都會被撈進來。

EchoLeak 真正的價值不在這一個漏洞被修好了，而在它證明了整條威脅鏈在真實的生產系統上是可以串通的。它正好對上 Willison 那個「致命三要素」的框架：當一個 agent 同時擁有存取私密資料、接觸不受信任內容、能把資料送出去這三隻腳，它就具備被注入後外洩的完整條件。Willison 給的解法也很冷酷——你沒辦法靠提示教會模型不上當，你只能去砍掉其中一隻腳。

最後還有一個最被低估的差異，就是放大。傳統的注入只攻擊模型的單一次輸出，可是 agent 有一個規劃迴圈，它會把任務拆成很多步、依序呼叫工具、再根據結果規劃下一步。所以一次成功的劫持，不是改變一個答案，而是重新導向整個計畫跟後續所有動作。多 agent 系統更糟，注入內容會沿著任務委派鏈往上爬，最後以權限最高的那個 agent 的身分執行。放大的終點，就是注入直接變成在主機上執行任意程式碼——微軟今年五月就揭露過一個 agent 框架的這種漏洞，一句提示就能在跑 agent 的機器上把計算機叫起來。

所以這一集的重點是：注入是入口，過度代理是放大器，自主多步執行是傳播路徑，工具跟權限是最後的爆炸當量。我們要防的，不是模型說錯話，而是防一個持有真實權限、會自主動手的系統，被外部文字劫持去執行攻擊者的意圖。知道威脅長這樣、也知道它沒有銀彈之後，接下來四集講的都是同一件事——在注入終將發生的前提下，把這個爆炸半徑一層一層收回去。下一集，我們就從架構開始。
