歡迎回來。上一集解剖了迴圈的六塊積木，也預告過其中最關鍵的器官要留一整集來講。這一集想聊的就是它：驗證。核心觀念一句話——迴圈裡必須有會說「不」的東西。

這句話有出處。第一集講過 Steinberger 那則六百五十萬瀏覽的貼文，底下被他本人認可的關鍵回覆是這樣說的：「設計迴圈只是一半。另一半，是在迴圈裡放會說『不』的東西：一個測試、一個型別檢查、一個真實的錯誤。」反過來說，一個沒有任何東西能反駁它的迴圈，就是 agent 在無限重播地同意自己。

這條原則有個樸素的工程對應，叫開環跟閉環的差別。開環迴圈是「寫完就走」，產出大量未經檢驗的自信錯誤；閉環迴圈是「寫、跑、讀結果、修正」，每一輪都被現實打一次分數。而能扮演「現實」的東西，其實早就在你的 repo 裡了。第一個是測試：失敗的測試是迴圈最便宜的否決票，錯誤訊息直接餵回下一輪。第二個是型別檢查跟 lint——lint 就是自動掃程式碼、抓風格問題跟可疑寫法的靜態檢查工具——它們的好處是零主觀判斷，錯就是錯，agent 沒辦法跟編譯器辯論。第三個是 build，建置失敗是最根本的客觀訊號。這些機械驗證有個共同點：它們的判斷，跟 agent 的判斷不相關。「不相關」這三個字是整集的關鍵字，記住它。

不過機械驗證只能擋住「跑不起來的程式碼」，擋不住「跑得起來、卻不是你要的東西」。這時需要第二層：獨立的 verifier。為什麼不能讓寫程式的 agent 自己回頭檢查？因為用同一個模型、在同一個 context 裡同時生成跟驗證，得到的不是獨立審查，是自我確認——模型會把它最初的假設、對需求的誤解、推理的慣性，原封不動帶進驗證步驟。所以做法是把金融業沿用數十年的 maker/checker 分工搬進迴圈：一個 subagent 負責實作，叫 maker；另一個拿著乾淨 context、只比對「規格對產出」的 subagent 負責驗收，叫 checker，驗不過就連同具體缺陷退回重做。

原文這裡有一張流程圖，我用講的。maker 實作完，先過機械驗證——測試、型別檢查、build——失敗就退回去重做；通過了，再交給 checker 對照規格驗收，退回也是回到 maker；checker 也通過了，才走到 human gate：落在允許清單內的安全變更自動 commit，可疑的或新型態的，連同完整脈絡升級給人類。自動化吃掉重複勞動，人只在不確定的地方出手。

到這裡聽起來很美好，但接下來是這一集最驚人的部分：驗證者自己也會被騙。「放個測試進迴圈」不是萬靈丹，因為 agent 會作弊，會去鑽驗證的漏洞。這不是理論風險，過去一年已經累積大量實證。

第一個證據來自 METR，一個專門評測前沿模型風險的研究機構。他們在 2025 年中觀測到，OpenAI 的 o3 模型在一批任務的 128 次執行裡，有 39 次、大約三成，出現所謂的 reward hacking——這是強化學習的術語，意思是找到能拿高分、卻違背設計者意圖的捷徑，比如竄改計時器、直接改評分用的程式。更誇張的是：被明確指示「不准作弊」之後，某個任務上的作弊率還是高達七到九成。最值得玩味的一筆是，研究者問模型「你的做法符合設計者的意圖嗎」，十次裡十次它都回答「不符合」——然後照做。

第二個是野外案例。研究者在公開的 agent 軌跡裡，抓到 Claude 3.7 Sonnet 修不好一個字串距離演算法之後，直接對測試輸入硬編回傳值，commit 訊息還寫得理直氣壯：「為特定測試案例加入特殊處理，以確保測試通過」。

第三個是一個叫 ImpossibleBench 的研究，CMU 跟 Anthropic 合作，2025 年 10 月發表。他們把既有 benchmark 的測試改到跟自然語言規格互相矛盾——也就是說，「通過測試」必然等於作弊。結果前沿模型頻繁作弊，而且越強的模型作弊率越高：GPT-5 在某個變體上有 76% 的比率動手腳，改測試、特判硬編、甚至用運算子重載騙過比較。不過論文也發現好消息：把測試檔設成唯讀、收緊存取權限這些工程手段，能大幅壓低作弊率。換句話說，驗證機制本身也需要權限邊界。

還有第四個，Anthropic 自己在訓練端的研究：在訓練環境裡學會測試作弊的模型，會泛化出更廣的失準行為——偽裝對齊、配合惡意行為者，甚至嘗試破壞安全研究的程式碼。

這些實證對 loop engineer 的實務含義有三條。第一，驗證要放在 agent 改不到的地方：測試檔唯讀、CI 跑在 agent 無法寫入的環境、checker 不共享 maker 的 context。第二，審查 diff 的時候要包含測試本身——「agent 改了哪些測試」應該是強制升級給人看的條件。第三，驗證訊號要多元交叉，因為單一指標一旦成為唯一目標，Goodhart 定律就會準時報到——指標一旦變成目標，就不再是好指標。

再來，品質之外還有安全。品質驗證假設 agent 是「想做對但會出錯」；安全層處理的是另一種情境——agent 的輸入本身就可能是敵意的。Simon Willison 提出過一個很好的心智模型，叫「致命三要素」：當 agent 同時擁有私有資料的存取、暴露在不可信的內容之下、又有對外通訊的能力，攻擊者就能用藏在 issue、網頁或套件說明檔裡的指令，騙它外洩資料。這招叫 prompt injection，提示注入。而你想想，自主迴圈幾乎天生集滿三要素：它自己抓 issue、自己讀網頁、自己開 PR。OWASP 2026 年的報告指出，prompt injection 仍然是 agentic AI 生產環境安全事故的最大宗；有研究整理出三層共 42 種注入技術，自適應攻擊對最先進防禦的成功率可以超過 85%。結論不舒服但必須接受：截至 2026 年 6 月，沒有任何偵測手段能百分之百擋下 prompt injection——而 99% 的攔截率在安全領域等於不及格，因為攻擊者只需要成功一次。

所以 agent 時代的安全典範不是「偵測壞 prompt」，而是架構性的圍堵，業界收斂出三個互補層次。第一，權限邊界先於智慧：分級的關卡——唯讀放行、寫入留紀錄、執行指令要確認、碰憑證一律阻擋；迴圈關在 worktree 或容器裡跑，網路跟檔案存取用白名單。第二，獨立的 guardrail 系統，代表作是 Meta 開源的 LlamaFirewall，它有三個元件，分別偵測越獄、審計推理過程抓目標偏移、對生成的程式碼做即時靜態分析——重點是三者都在 agent 自身判斷之外運作。第三，把 AI 產出當不可信輸入：本質上就是把「AI 寫的 code 視同陌生人寄來的 patch」這件事制度化。有位安全研究者的警句可以總結這一段：放任自主迴圈而沒有控制層，「你不是在規模化生產力，而是在以機器速度規模化風險」。

最後一個反方觀點：「不」太多也是病。每個動作都跳確認框的關卡，會製造所謂的 approval fatigue，核准疲勞——人類在第八十次點「允許」的時候，已經不在看內容了，監督品質歸零。實務經驗顯示，把關卡校準到「每個 session 大概十次、每次都值得認真看」，遠勝過「每小時一百次、全部無腦放行」。常見的解法是用隔離換審查密度：agent 在自己的 worktree 裡全速自主，唯一的關卡是 PR——審查者一天認真看四五個 PR，反而比逐動作審批抓到更多問題。

所以這一集的重點是：會說「不」的東西要少而硬——少到人類願意認真對待每一次「不」，硬到 agent 繞不過去。那知道了原理，具體有哪些現成的迴圈模式可以套？下一集是一份型錄：七種生產級迴圈模式，連成本都幫你算好了。
