歡迎回來。上一集講了迴圈裡要有會說「不」的東西，這一集輕鬆一點，是一份型錄：截至 2026 年 6 月，社群已經沉澱出七種可以直接套用的迴圈模式。我會把每一種解決什麼痛點、多久跑一次、燒多少 token 講給你聽，最後給你一套選型的規則。

先把全貌講一遍。原文有一張表，我用講的，順序從便宜到貴。另外先解釋兩個分級：L1 是 report-only，只看、只提案、不動手改 code；L2 是可以在受限範圍內動手修。

第一種，Daily Triage，每日巡檢。解決的痛點是你每天早上手動巡 CI、巡 issue、巡 PR、到聊天室問「哪裡著火了」。建議一天跑一次，L1 只出報告，成本低。第二種，PR Babysitter，PR 保母。解決的是 push 之後反覆等 CI、等 review、修一點再等的碎片化等待。節奏很密，五到十五分鐘一次，成本高。第三種，CI Sweeper，CI 清道夫。解決的是主幹或活躍分支的測試紅燈沒人第一時間處理。一樣五到十五分鐘，但它是 L2 會動手修的，成本是七種裡最貴的，極高。第四種，Dependency Sweeper，依賴清掃。解決 Renovate、Dependabot 這類自動更新工具開的 PR 堆著沒人看的問題，六小時到一天跑一次，L2 但只准碰 patch 等級的小版本，成本中等。第五種，Changelog Drafter，更新日誌起草。解決 release notes 永遠最後一刻趕工的問題，一天一次或發版時觸發，L1 出草稿，便宜。第六種，Post-Merge Cleanup，合併後清理。掃 merge 之後殘留的暫時 log、沒用到的 import、死碼，離峰時段跑，便宜。第七種，Issue Triage，議題分類。新 issue 沒人標 label、重複 issue 沒人關，兩小時到一天跑一次，L1 只提案，便宜。

從這張型錄可以讀出兩個觀察。第一，節奏跟成本強相關，但不是同一回事：PR Babysitter 跟 CI Sweeper 都是五到十五分鐘的節奏，後者卻貴得多，因為「修一個紅燈 CI」平均要動的脈絡，遠大於「回一條 review 意見」。第二，七種裡有五種建議從 L1 起步——社群共識很明確：先讓迴圈證明它看得準，再讓它動手。

接下來幫高頻雙雄單獨算帳，因為它們回報感最強、也最容易燒錢。

先看 PR Babysitter 的單次成本結構：醒來發現沒事做的那一輪，叫 no-op，大概三千個 token；完整 triage 一次大概八萬；含驗證的修復嘗試大概二十五萬。以十分鐘節奏跑，建議日上限兩百萬 token。它成立的前提是「大多數輪次是 no-op」——監看清單空了就秒退。實戰面最常被引用的案例，是 Jökull Sólberg 公開的 babysit-pr 技能，大概一百七十行 markdown：平行輪詢 CI 跟 AI review 工具的意見，把所有 feedback 分成三類——修掉、駁回、升級給人——修完 push，循環到 PR 乾淨為止。從 push 到可以 merge 大概十分鐘，其中大半是在等 CI。關鍵設計是 triage 那一步：agent 對每條 review 意見做判斷，而不是照單全修，因為 review 工具的誤報率不低。

CI Sweeper 更貴：綠燈 no-op 約五千、triage 約五萬、完整修復嘗試約二十萬 token。十五分鐘節奏下，如果缺了「CI 是綠的就立刻退出」的早退邏輯，一天可以燒超過五百萬 token。它的典型風險叫「修症狀迴圈」——把 CI 弄綠了，但沒修到根因。所以這個模式強制要求獨立 verifier 把關，原理就是上一集講的；而且同一個失敗修三次沒成、或者要動超過五個檔案、要碰核心系統，就必須升級給人。還有一條鐵則：flaky 測試多的 codebase 不要跑這個模式。flaky 測試就是程式碼沒動、卻時好時壞的不穩定測試——迴圈會把 token 燒在追逐幽靈上。

不過，這類「掃失敗、派工修」的思路，在基礎設施到位時上限很高。Brian Douglas 在 2026 年 3 月做過一次實測：用一個叫 Sweeper 的工具，把 842 個 ESLint 錯誤、橫跨 99 個檔案，按檔案分組、生成 prompt、派給五個平行的 Claude Code 子代理。三輪、總共 54 分鐘、百分之百修復率，最終的 PR 改了 110 個檔案。他的結論不是「prompt 寫得好」，而是排程、分組、記錄每輪策略跟 token 成本的基礎設施，才是能力的上限。

再來講四種被低估的便宜模式。它們的共同點是讀多寫少，輸出是報告、草稿或標籤，不是程式碼變更，所以單次成本低、出錯代價也低。Daily Triage 是官方推薦的入門模式：每天早上掃一輪，產出一份排好優先序的狀態檔，先 report-only 跑一兩週，人類確認訊號可信之後，才考慮加上小型修復能力。Issue Triage 是實戰案例最多的一型，有人用大概五百行 TypeScript 寫成 GitHub Action：新 issue 進來，模型分類上標籤，然後做兩段式的重複偵測——先用便宜的 GitHub 原生搜尋收斂出候選名單，貴的 LLM 只看決賽圈。這個兩段式設計就是典型的成本控制。另外 Metabase 的做法展示了另一種取捨：他們刻意不讓 bot 全自動跑，人類先審過 issue 再觸發 agent 重現 bug，理由就是防上一集講的 prompt injection。Changelog Drafter 跟 Post-Merge Cleanup 都便宜到幾乎沒有「不跑」的理由——前提是你的團隊真的有這兩個痛點。Dependency Sweeper 介於中間：它比 Renovate 多的是讀 changelog 跟遷移文件、再動手改 code 的能力，但起手務必鎖定 patch-only，大版本升級的爆炸半徑不是 L2 該碰的。

最後，把這些帳整合成四條選型規則。第一條，從你每天真的在浪費時間的地方選，不是從最酷的選。每天早上花二十分鐘巡狀態，就上 Daily Triage；issue 進來三天沒人標，就上 Issue Triage；PR 平均要 push 四次才綠，才考慮 PR Babysitter。第二條，節奏跟著事件密度走。五到十五分鐘的節奏，只有在事件真的每小時都在發生時才划算；一天兩個 PR 的 side project 跑 PR Babysitter，就是付高頻的錢、買低頻的事。寧可從一天一次開始，嫌慢再加密。第三條，每種模式都要設 token 日上限跟早退條件。no-op 必須便宜，三到五千 token 的等級，這是所有模式共同的生死線。有分析指出 multi-agent 系統的 token 消耗大約是單純聊天的十五倍；連 GitHub 自己跑 agentic workflow，都得靠工具修剪跟每日稽核 agent，才把 token 開銷砍下最多 62%——連 GitHub 都要事後減肥，你最好事前節食。第四條，L1 到 L2 到 L3 分階段，每階段至少觀察一週。全部先 report-only，誤報率可接受再開放動手；無人值守是所有驗證閘門都通過之後的事，不是第一天的事。

收尾前一個反方提醒：這七種模式，沒有一種能拯救「事件密度低、加上 CI 不穩定」的專案。flaky 測試會讓 CI Sweeper 空轉燒錢，低流量會讓高頻模式的固定成本攤不掉——這種時候，正確答案是先修 CI，或者乾脆不跑迴圈。

所以這一集的重點是：選模式不是選最炫的，是選你每天真的在流血的地方，從便宜的、只報告的開始，讓迴圈先證明自己看得準。那型錄看完手癢了嗎？下一集就是動手做：從五行的 shell 腳本開始，四個階梯，搭出你的第一個迴圈。
