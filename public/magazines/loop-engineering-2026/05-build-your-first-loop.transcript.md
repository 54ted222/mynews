歡迎回來。前面四集講了迴圈怎麼來的、由什麼組成、怎麼驗證、有哪些模式，這一集只回答一個問題：指令到底怎麼打。我們會沿著四個階梯往上爬：從一段 bash 腳本，到官方的 Ralph Wiggum 外掛，到 session 裡的 /loop，最後到 GitHub Actions 上完全無人值守。先把總綱講在前面：一個能上生產的迴圈，只需要四樣東西——重複執行的機制、迭代上限、迴圈外的驗證、明確的收工條件。每一階我都會標出「驗證放這裡」「上限設這裡」。還有一條鐵律：永遠先設上限，再開迴圈。

第一階，一段 bash 就是迴圈。Geoffrey Huntley 原始的 Ralph loop 真的只有一行：無限迴圈，每一圈把 PROMPT.md 的內容灌給 Claude，跑完再來一圈。狀態全部活在那份 prompt 檔、改過的檔案跟 git 歷史裡——每一輪都是全新的 context，agent 靠讀檔案接續前一輪的進度。這在 2026 年依然可用，但裸跑等於把信用卡插進飲料機。

所以實際可用的版本要加三樣東西，我用講的描述這個腳本在做什麼。第一步，先開一個 git worktree——就是同一個 repo 的另一個工作目錄——讓迴圈在隔離的環境跑，搞砸了也不污染你的工作目錄。第二步，把無限迴圈改成有上限的 for 迴圈，比如最多二十輪，沒有這個上限，失敗的任務會無限燒 token。第三步，每一輪結束後，在迴圈外面、用 shell 層的 if 判斷：跑 npm test 跟 npm run lint，全綠就宣告完成、退出；二十輪跑完還沒綠，就報錯收場。呼叫 Claude 的部分用的是 claude 加 -p 參數，這是非互動的單發模式，俗稱 headless，跑完就退出；再用 allowedTools 參數白名單限制它只能讀檔、改檔、跑測試跟 git 指令。有個查證過的拼寫陷阱順便講：白名單用的是前綴比對，星號前面要留空格，不然你允許 git diff，會連 git diff-index 這種指令一起放行。

這一階最重要的觀念是驗證的位置：npm test 跑在 shell 層，不是寫在 prompt 裡叫 agent「記得測試」。agent 在 prompt 裡可以自我感覺良好，但它改不掉你 shell 裡的 if。這就是第三集說的「會說不的東西」在實作上的最小形式。

第二階，不想自己管 bash，Anthropic 把這套收進了官方外掛。先用 /plugin 指令掛上官方 marketplace，再安裝 ralph-wiggum 這個 plugin。啟動方式是打 /ralph-loop，後面接你的任務描述，比如「建一個 todos REST API，要有增刪查改、輸入驗證、測試，全部測試通過後輸出 COMPLETE 這個承諾字串」，再帶兩個參數：completion-promise 指定收工字串，max-iterations 設上限，比如三十。要中途喊停就打 /cancel-ralph。

它的原理是 Stop hook：Claude 每次想結束回合，hook 攔下來、把同一份 prompt 原封不動餵回去；前一輪改的檔案跟 git 歷史都在，所以 agent 看得到自己的進度。收工條件只有兩個：輸出跟 completion promise 完全一致的字串——注意是精確比對，不能搞「成功或卡關」兩種結局——或者撞到迭代上限。

兩個實戰提醒。第一，官方文件明講，completion promise 是成功訊號、不是安全機制，max-iterations 才是你的保險絲——而且真的有人踩過保險絲失靈的案例：有 issue 回報過上限沒被遵守、整個月的用量被吃光。所以跑大任務之前，先用小上限、比如五輪，做一次冒煙測試。第二，在 prompt 裡留逃生口，比如寫「如果十五輪後還卡住：記錄阻礙、列出已嘗試的方案、建議替代做法，然後輸出承諾字串」。適用範圍也要講清楚：這招適合有自動驗證的明確任務——有測試、有 lint、或是從零開始的新專案；別拿去做需要人類判斷的事，或者線上環境的除錯。

順帶一提，Stop hook 也可以自己寫：在設定檔裡掛一個 Stop hook，腳本跑你的測試，沒過就回傳一個「擋下、原因是測試還在失敗」的 JSON，Claude 就無法收工，會帶著原因繼續修。這是「驗證放這裡」的第二個位置：擋在出口上。

第三階，/loop。前兩階是「不達標不准停」的衝刺型迴圈，/loop 是另一種——固定間隔重跑一個 prompt 的巡邏型。用法像這樣：打 /loop、間隔五分鐘、後面接「檢查 deploy 是否完成，失敗就把 log 撈出來診斷」；prompt 也可以是另一個斜線指令，比如每二十分鐘跑一次 review-pr。幾個查證過的行為細節：間隔支援秒、分、時、天；省略間隔的話，Claude 每輪後自己選一分鐘到一小時的延遲；任務是 session 範圍的，關掉終端機就停；所有迴圈七天自動到期；等待中按 Esc 就停。你還可以把預設的巡邏指令寫成一個 loop.md 檔，裸打 /loop 就會執行它。那這一階的上限在哪？七天到期、加上 session 必須開著，這兩道就是天然上限。驗證在哪?寫進 prompt 的動詞裡——寫「檢查 CI 是否通過」，而不是「處理 CI」，讓每一輪都從讀取客觀狀態開始。

第四階，GitHub Actions，無人值守。/loop 要求 session 開著；要機器關了也照跑，就把迴圈搬上 CI。最快的路徑是在 Claude Code 裡執行 /install-github-app，它會幫你裝好 GitHub App、設好 API key。然後一個用 schedule 觸發的 workflow 就是一個每日迴圈，我描述一下這份 YAML 做的事：每天 UTC 19 點、也就是台北凌晨 3 點觸發；workflow 層設 30 分鐘的 timeout，這是上限之一；步驟裡先 checkout 程式碼，再呼叫官方的 claude-code-action，prompt 寫「跑 npm test 跟 lint，有失敗就修到全綠、開 PR 說明改了什麼；全綠就只留一行報告」；最後透過 claude_args 傳 max-turns 15 限制推理回合數，這是上限之二。那驗證放哪?就放在你既有的 CI 上——Claude 開的 PR 一樣要過分支保護跟必要檢查，人類 merge 是最後一道閘。

不想用現成的 action，也可以在任何 CI 直接呼叫 claude -p。官方建議腳本場景加 --bare 參數，跳過本機的 hooks 跟 skills，換來每台機器結果一致。還有一個時效提醒：官方公告從 2026 年 6 月 15 號起，訂閱方案的程式化用量改計入獨立的月度額度，不再跟互動用量混算——排 CI 迴圈前先確認額度。

所以這一集的重點是：四階爬完，你手上其實是同一個東西的四種形態——重複、上限、外部驗證、收工條件。bash 給你完全掌控，plugin 給你現成護欄，/loop 給你巡邏節奏，Actions 給你無人值守。但無人值守聽起來很美，帳單跟風險也跟著來——下一集，整輯的最後一集，我們來算帳：有人一個晚上燒掉六千美元，還有一筆比錢更難還的債。
