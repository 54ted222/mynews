歡迎回來。上一集我們聊了 loop engineering 怎麼在十三個月內從笑話變成共識，這一集要回答一個更基本的問題：一個迴圈，到底是由什麼組成的？這一集是解剖學，我們會把一個生產級的迴圈拆成六塊積木，再把它們組回一條完整的流程。

先從定義開始。社群目前最常被引用的工作定義是這樣的：一個迴圈會發現工作、把工作派給 agent、驗證結果、把狀態持久化、然後決定下一步——依排程重複，或者跑到目標達成為止。Boris Cherny 的版本更白話：「我不再 prompt Claude 了，是迴圈在 prompt Claude、自己決定要做什麼。我的工作是寫迴圈。」

這個定義立刻劃出一條跟 cron job 的界線。cron，就是 Unix 從 1970 年代就有的排程工具，時間到了就執行同一份腳本，每次走同一條路徑。而 loop 是：時間到了，或者事件來了，由一個模型看著當前狀態，決定這一輪要做什麼，做完還會檢查有沒有做成，再決定要繼續、換方向、還是收工。一句話總結：loop 等於 cron 加上一個決策者。排程只是觸發器，真正的差異在中間那顆會評估、會調整的腦。當然，「這顆腦值不值那個 token 錢」是另一個問題，我們留到第六集算帳。

還有一個容易混淆的點：loop engineering 不會取代 context engineering。每一輪迴圈醒來的時候，還是要把對的檔案、對的歷史、對的工具定義放到模型面前。等一下講的六塊積木裡面，至少有三塊本質上就是「跨輪次的 context engineering」。

好，進入正題，六塊積木。每一塊我都會講它解決什麼問題、缺了會怎樣。

第一塊，排程與自動化。它解決的是「誰來啟動這一輪」。沒有它，發現工作、分派工作的還是你本人，那只是把 prompt 寫長一點，不是迴圈。觸發器可以是固定排程，也可以是事件：PR 開了、CI 轉紅了、Slack 訊息來了。以 Claude Code 來說，官方就提供三層：雲端的 Routines，跑在 Anthropic 的基礎設施上，機器關了也照跑；桌面版的排程任務；還有 session 裡的 /loop 指令。有個細節很值得玩味：/loop 的排程刻意加了最多三十分鐘的隨機延遲，而且七天自動到期——官方文件自己就在防「被遺忘的迴圈永遠跑下去」，這是這塊積木成熟的標誌。

第二塊，git worktree。worktree 就是 git 讓你同一個 repo 同時開好幾個工作目錄的功能，共享歷史，但各自有自己的分支跟檔案。它解決的是「兩個 agent 同時動同一份 checkout 會互踩」——共用目錄的下場很好預測：互相覆寫、context 錯亂、build 壞掉。缺了這塊，你的迴圈只能一個接一個跑，或者在某個深夜學到並行寫檔的代價。

第三塊，skills，持久的專案知識。一個 skill 就是一個資料夾加一份說明檔，裡面寫清楚什麼時候觸發、步驟是什麼。重點是用「具名的、可重複呼叫的食譜」取代一次性的 prompt：triage 怎麼做、修 CI 的標準程序是什麼，寫成 skill 之後每輪行為一致，還可以 review、可以進版控。缺了這塊，迴圈每輪的行為會飄移，你會看到同一個問題在不同迭代被三種不同方式「解決」。

第四塊，MCP 連接器。MCP 是一個開放協定，把 agent 接上 Git 平台、ticket 系統、Slack 這些外部世界。它解決的是「迴圈做完的事怎麼落地」。沒有它，迴圈就是個沙盒：分析得再對，也只能寫進本地檔案，開不了 PR、回不了 issue，成果出不了迴圈本身。

第五塊，subagent 分工，也就是 maker 跟 checker。subagent 是主 agent 派出去的子代理，各自有獨立的對話脈絡跟工具權限，中間過程不回流，只有最終結論回來。這個隔離正是分工的基礎：驗證的人不繼承實作的人的假設、脈絡跟盲點。它解決的問題是「同一顆腦自己寫、自己改、自己說沒問題」。缺了這塊，迴圈就變成一台自我同意的機器。這是整張解剖圖裡最關鍵的器官，這集只標出位置，下一集整集都在講它。

第六塊，外部狀態。要記住一件事：context window 不是記憶體，是暫存緩衝。真正的記憶必須活在模型之外——磁碟上的狀態檔、ticket 系統、或者知識圖譜。Ralph 系迴圈的核心架構正是如此：每一輪用全新的 context 重啟 agent，狀態靠檔案往前傳。缺了這塊，迴圈沒有連貫性：同一件事重做三次、做過的決策被推翻。六塊裡面，這是新手最常省略、也最快讓迴圈崩潰的一塊。

那這六塊組起來長什麼樣？原文有一張流程圖，我用嘴巴帶你走一輪。首先，排程或事件把迴圈喚醒；第一站是 triage，就是檢傷分類——它先讀外部狀態，看上一輪做到哪、哪些事擱置中，再掃描現實：issue、CI、PR，挑出這一輪值得做的工作。這是第一個決策點，cron 沒有的那種。接著，對每項工作開一個隔離的 worktree，派 implementer subagent 去實作；做完之後，第二個 subagent 拿著專案的 skills 跟既有測試，獨立審查那份草稿。通過驗證的成果，走到一個叫 human gate 的關卡做條件分流：被標記為安全的、在允許清單內的，自動 commit 或開 PR；有風險或模稜兩可的，帶著完整脈絡升級給人——不是丟一句「請看一下」，而是附上做了什麼、為什麼不確定、驗證結果如何。最後，把這一輪的結論寫回狀態，迴圈重新開始。

這條流程有兩個地方值得注意。第一，狀態讀寫出現在頭尾兩端：開頭讀，決定做什麼；結尾寫，記住做了什麼。這條雙向的連結，就是迴圈跟「定時跑的腳本」在結構上的差別。第二，human gate 不是裝飾。原作者自己的警語是：驗證仍然是你的責任，無人看管的迴圈會犯無人看管的錯。

最後一個實際的問題：哪些積木可以先省？誠實的答案是，看你要玩具還是生產系統。最小可行的迴圈只需要兩塊：觸發器和外部狀態檔，這就是 Ralph loop 的原始形態。但每省一塊，就接受一種退化：沒有 worktree 不能並行；沒有 skills 行為飄移；沒有 MCP 成果出不了本機；沒有 checker，你得到一台高速自我同意機。實務上的擴張順序通常是：先有觸發加狀態讓迴圈能轉，再補 checker 讓它值得信任，最後才加 worktree 跟 MCP 讓它規模化。

所以這一集的重點是：迴圈不是「定時跑的 prompt」，而是排程、隔離、知識、連接、分工、記憶六塊積木組成的系統，中間有一顆會做決策的腦。而六塊裡最關鍵的那一塊——會說「不」的 checker——下一集我們整集來講：為什麼沒有獨立驗證的迴圈，只是一台快速生產自信錯誤的機器。
