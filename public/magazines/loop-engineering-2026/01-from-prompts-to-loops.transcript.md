哈囉，歡迎回來。導聆裡我們說，這一期要從那個被全世界嘲笑、最後笑到最後的 while true 開始講起。所以這一集想聊的就是：loop engineering 這個概念，是怎麼在短短十三個月之內，從一個笑話變成整個 AI 開發圈的共識的。

先把一件事說清楚：「讓語言模型在迴圈裡跑」根本不是 2026 年的新點子。早在 2022 年 10 月，就有一篇叫 ReAct 的學術論文——ReAct 是 Reason 加 Act 的縮寫，意思是讓模型先想一步、做一個動作、看環境給的回饋，然後再進下一輪。這個「思考、行動、觀察」的循環，後來變成幾乎所有 coding agent 的內部骨架。換句話說，每次你按下 Enter，agent 肚子裡早就在跑迴圈了。

那 2026 年講的 loop engineering 到底新在哪？新的是迴圈的層級。剛剛說的那種迴圈，是 agent 做「一次任務」的內迴圈，由模型供應商寫死在產品裡；而 loop engineering 講的是包在外面的外迴圈——誰來決定 agent 下一個任務是什麼？做完了沒？狀態存在哪裡？什麼時候該停？這一層過去一直是人在扮演的：你開一個 session、貼 prompt、看結果、再貼一次。這十三個月發生的事，簡單說，就是這層「人肉外迴圈」被一步一步寫成了程式。

故事的實務起點有明確的人名。澳洲工程師 Geoffrey Huntley 在 2025 年上半年發現一件事：與其花時間精修 prompt，不如把同一份 prompt 檔無腦地、反覆地餵給 agent，讓它每一輪自己去讀磁碟上程式碼的現狀，然後繼續往前推。他公開的原始版本粗暴到近乎挑釁——就是一行 shell 指令：無限迴圈，每一圈把 PROMPT.md 的內容灌給 agent，跑完再來一圈。就這樣。

連名字都很自嘲。Ralph Wiggum 是辛普森家庭裡那個一邊撞門框一邊喊「我在幫忙！」的傻孩子；而 ralph 這個字在俚語裡是嘔吐的意思。Huntley 自己受訪時說，他在 2025 年 2 月意識到這招居然有用的時候，「想吐」。他在舊金山一場大概十五個人的小聚現場 demo，還在自己睡覺的澳洲深夜開直播，讓 agent 通宵寫程式；同年 7 月發了那篇後來成為正典出處的部落格文章。

當時主流的反應是什麼？訕笑跟不信。批評集中在幾點：沒有狀態管理、沒有驗證、純靠暴力重試燒 token。技術論壇 Hacker News 上甚至到 2026 年初還有標題很直白的討論串，叫「總得有人說出來：Ralph Wiggum 根本行不通」。批評者說，正解應該是把任務拆解成步驟、在步驟之間傳遞明確的狀態、配上驗證跟護欄——而不是把一切丟進一個更大的迴圈。但這裡有個很有趣的轉折：這些批評後來大多沒有否定掉迴圈本身，反而變成了 loop engineering 的設計清單。今天大家講的驗證器、外部狀態、實作跟驗收分工，幾乎是逐條回應當年的嘲諷。

而 Ralph 能活下來，靠的不是辯論，是產出。Huntley 用它做出了一個叫 Cursed 的程式語言，編譯器是迴圈自己寫的；做了一個給 1980 年代老電腦 ZX Spectrum 用的報稅應用；後來甚至逆向複刻了一個商用產品。科技媒體 The Register 在 2026 年 1 月底的報導標題說得很露骨：Ralph 迴圈能用每小時 10 美元的代價「氛圍複製」商用軟體，而且不少 Y Combinator——就是矽谷最有名的新創加速器——旗下的新創已經在用這招了。

接下來是轉折點：官方收編。2025 年 12 月中，Anthropic 把 Ralph Wiggum 做成 Claude Code 的官方外掛。官方版把那個 bash 外迴圈內化成 session 裡的機制：用一個叫 Stop hook 的東西——就是在 agent 想收工的時候攔截它、把同一份 prompt 重新餵回去；再加上最大迭代次數的安全上限，跟一個「完成承諾」字串，agent 輸出那個精確字串才算真的做完。收編的意義不只是背書，而是平台方等於承認了一件事：外迴圈是產品該管的事，不是使用者自己搞的 hack。然後 2026 年 4 月，Anthropic 再開放 Routines 的預覽：prompt、repo、連接器設定一次，迴圈就在雲端按排程自動跑，你筆電闔上也照跑。基礎設施備齊了，只差一個名字。

名字在五週內定調。2026 年 5 月 6 日，Code with Claude 大會上，Claude Code 的創造者 Boris Cherny 壓軸演講，現場 demo 自治 agent 處理真實的 GitHub issue，然後說了那句流傳很廣的話：「我不再 prompt Claude 了。我有迴圈在跑、在 prompt Claude、在決定要做什麼。我的工作是寫迴圈。」一個月後，6 月 8 日，Peter Steinberger 在社群平台發文：「每月提醒：你不該再親自 prompt coding agent 了，你該設計會去 prompt 你的 agent 的迴圈。」據報導衝出大概六百五十萬瀏覽。Steinberger 的話特別有重量，因為他的履歷橫跨兩大陣營——他做的開源 agent 專案曾經在三個月內累積近二十萬顆星，2026 年 2 月他加入了 OpenAI。當 Anthropic 生態的 Cherny 跟 OpenAI 的 Steinberger 在一個月內說出幾乎同一句話，這就不是哪家廠商的行銷了，是跨陣營的從業者共識。

最後聊聊「為什麼是現在」。把十三個月攤開看，這其實是一場槓桿點的接力。第一棒是 prompt engineering，怎麼問好一句話；第二棒是 2025 年 6 月 Karpathy 背書的 context engineering，重點變成怎麼把對的資訊組裝進模型的工作記憶；到了 2026 年，context engineering 又被收編成迴圈裡的一個元件——你設計的迴圈，每一輪都在替 agent 做 context engineering。而第三棒能在現在交接，是因為三個條件同時成熟了：第一，模型能力夠強，一輪迴圈能消化的工作量從「改一個函式」變成「修完一張 issue 含測試」；第二，工具成熟了，以前要自己拿 bash 黏的東西，現在都是平台原生零件；第三，驗證閉環——迴圈要無人值守，前提是有測試、型別檢查、CI 這些會自動說「不」的東西把關。

所以重點是：Ralph Wiggum loop 在 2025 年被嘲笑，不是因為方向錯，而是它把一個三年後才會成熟的架構，用最裸露的方式提前演了出來。嘲笑的人看到的是沒有護欄的無限迴圈；Huntley 看到的是，當模型夠強、工具夠全，工程師的位置必然從迴圈裡面移到迴圈外面。現在平台、命名、共識都到位了，剩下的問題不再是「要不要寫迴圈」，而是「迴圈該長什麼樣子」。下一集，我們就來做解剖——一個生產級的迴圈，到底由哪六塊積木組成。
