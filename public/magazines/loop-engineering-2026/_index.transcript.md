哈囉，歡迎收聽。這一集是本期雜誌的導聆，主題只有一個詞：loop engineering，迴圈工程。如果你最近有在關注 AI 開發圈，你大概躲不掉這個詞。

事情是這樣開始的。2026 年 6 月 8 號，一位很有影響力的工程師 Peter Steinberger 在社群平台上發了兩句話，意思是：你不應該再親自去 prompt 你的 coding agent 了，你應該去設計那個會去 prompt agent 的迴圈。就這麼兩句話，據報導衝出了六百五十萬次瀏覽。而且有趣的是，一個月前，Claude Code 的創造者 Boris Cherny 在大會上也講了幾乎一模一樣的話，他說：我不再 prompt Claude 了，是迴圈在 prompt Claude，我的工作是寫迴圈。兩個分屬不同陣營的指標人物，在一個月內說出同一句話，這個概念就這樣一夜之間變成了 AI 開發圈的新共識。

不過，這不是憑空冒出來的流行語。往回追一年多，有位澳洲工程師 Geoffrey Huntley，在 2025 年上半年做了一個粗暴到近乎挑釁的實驗：他用一行最簡單的無限迴圈指令，把同一份提示檔反覆餵給 AI agent，讓它自己一輪一輪往前做。這個東西叫 Ralph Wiggum loop，名字取自辛普森家庭裡那個傻傻的角色。當時整個社群都在嘲笑它，說這蠢得可以、純粹是燒錢。結果呢？最後它被 Anthropic 收進了 Claude Code 的官方外掛。一個被嘲笑的笑話，變成了官方功能。

所以你可以把這個演變想成三個世代。第一代叫 prompt engineering，重點是怎麼問好一句話；第二代叫 context engineering，重點是怎麼把對的資訊餵進模型的工作記憶；而現在的第三代 loop engineering，重點變成：怎麼設計一個系統，讓它自己會發現工作、自己派工、自己驗證、自己記住狀態、自己決定下一步。簡單說，工程師的位置，從迴圈裡面，移到了迴圈外面。

那這一期六篇文章，是沿著一條主線走的。我快速帶你走一遍。

第一篇講歷史，就是我剛剛說的那個故事的完整版：從學術上 2022 年的研究論文，到被嘲笑的 Ralph Wiggum loop，到官方收編，再到 2026 年 6 月正式定名，十三個月的完整系譜。

第二篇是解剖學。一個生產級的迴圈到底由哪些零件組成？答案是六塊積木：排程、隔離的工作目錄、可重複使用的專案知識、對外連接的通道、分工合作的子代理，還有最重要的，活在對話之外的外部狀態。這一篇也回答一個常見的疑問：這跟一個定時排程的腳本到底差在哪？

第三篇是全期最關鍵的一篇，講驗證。核心觀念一句話：迴圈裡必須有會說「不」的東西。一個沒有獨立驗證的迴圈，只是一台快速生產自信錯誤的機器。這一篇也會講到很驚人的研究發現：AI 其實會作弊，會去鑽驗證機制的漏洞，所以驗證本身也要設防。

第四篇很實用，是一份型錄：社群已經沉澱出七種可以直接套用的迴圈模式，從幫你照看 PR 的保母，到自動修測試紅燈的清道夫，每一種的成本、節奏、風險都幫你算好了。

第五篇是動手做。如果你聽完前面四篇心癢了，這一篇手把手教你，從五行的 shell 腳本開始，一路爬到完全無人值守的雲端迴圈，四個階梯。

最後第六篇，回頭算帳。迴圈不是免費午餐：有人一個晚上燒掉六千美元；有人質疑這整套東西只是老掉牙的排程工具換個新包裝；還有一個更深的問題，叫理解債——當程式碼出貨的速度快過任何人理解它的速度，工程師的角色還剩下什麼？

所以這一期聽完，你應該能回答三個問題：第一，loop engineering 到底新在哪？第二，一個值得信任的迴圈長什麼樣子？第三，你自己的第一個迴圈，該從哪裡開始？
