歡迎來到本期的最後一集。前五集講了迴圈怎麼來的、怎麼拆、怎麼驗、怎麼套模式、怎麼動手做——這一集，我們回頭算帳。一共三本帳：第一本是看得見的 token 帳單；第二本是這個概念本身欠不欠「新意」的帳；第三本最難還，叫理解債。

先算最直接的。迴圈的浪漫版本是「你睡覺時一千個 agent 幫你蓋產品」；生產版本是「你大部分的工程時間花在確保它們會停」。這個落差，最先用錢的形式顯現。

公開的帳單事故不少。最有名的一個：有位 Reddit 用戶讓 Claude Code 每三十分鐘檢查一次軟體更新，然後去睡覺——醒來收到六千美元的帳單。這個案例值得細講，因為成因很有教育意義。Anthropic 在沒有公告的情況下，把 prompt cache 的預設存活時間從一小時改成五分鐘。prompt cache 是服務端的快取機制，對話前段的運算結果暫存起來，後續請求可以大幅降價；但快取過期就要付全價重建。而這個迴圈每三十分鐘才醒一次，每次醒來快取都已經死了，導致系統一天四十八次、從零重建大概八十萬 token 的對話歷史——而且「寫入快取」遠比「讀取快取」貴。注意這個案例的關鍵：這位開發者沒有亂用工具，他完全照著「自主、隔夜、最少監督」的宣傳方式在用，帳單照樣失控。另一起案例：一位高階訂閱方案的用戶照著建議排程跑隔夜任務，兩天內被收一千八百美元的 API 費用。還有更極端的多 subagent 事故，金額據稱上看數萬美元，當量級參考就好。

結構性的問題有兩個。第一，計費的回饋迴路比 agent 的迴圈慢：沒有即時的消費計數器，用量儀表板有數天延遲——對凌晨三點在跑的無人值守迴圈來說，事後告警毫無意義。第二，成本隨架構超線性放大：每多派一個 subagent、每多跑一輪，帳單不是線性加總，因為每個分身都帶著自己的 context 在燒。所以 2026 年中所有嚴肅的迴圈論述，都收斂到三道不可妥協的硬煞車：最大迭代次數，不管迴圈自認有沒有進展都強制停；無進展偵測，發現在原地打轉就先停；還有 token 或美元的預算上限，超標就殺掉迴圈。

第二本帳，懷疑論。Steinberger 那則六百五十萬瀏覽的貼文底下，最鋒利的一句回覆是：「cron job 最近的改名挺有趣的嘛。」意思是，你們講得天花亂墜的 loop engineering，不就是老掉牙的排程工具換個新包裝？這句話值得認真對待，因為它對了一半。

對的那一半：排程層確實就是 cron。連 Boris Cherny 自己的迴圈都是用 cron 排的。如果你的「迴圈」只是每小時把同一份 prompt 丟給模型一次，那它確實就是一個 cron job——而且 cron 比在場大多數人的年紀都大。更尖銳的批評說：把一串機率性的 prompt 用迴圈鏈起來，不叫產品，叫「以 token 計價的官僚體系」——持續產生帳單，同時把漂移藏在自動化的表象底下。

錯的那一半：cron 跟 loop 的差異不在排程，在迴圈本體裡的決策者。這是第二集講過的——cron 執行固定腳本，每次走同一條路；loop 裡跑的是一個 agent，它檢視當下狀態、判斷需不需要動手、選擇下一步、執行、檢查結果、再決定接下來。Cherny 在生產環境跑的那些迴圈——照看 PR 的 rebase 跟 CI 修復、找出並修補不穩定的測試、每三十分鐘聚類一次社群回饋——沒有一個能用固定腳本寫出來，因為每次醒來「該做什麼」都不一樣。誠實的裁決是：「cron 換皮」描述了地基，但漏掉了房子。支持方自己也承認排程層毫無新意；他們主張的新東西，是包在決策外面那一圈——狀態檢查、範圍限制、停止條件、獨立驗證。而懷疑論真正的貢獻是提醒你：如果你的迴圈沒有那一圈，它就真的只是一個比較貴、比較不可預測的 cron job。

第三本帳最深。token 帳單至少看得到數字；Addy Osmani——Google Chrome 團隊的資深工程主管——提出的 comprehension debt，理解債，量化起來困難得多。定義是：系統裡存在多少程式碼，跟任何一個人類真正理解多少程式碼，這兩者之間不斷擴大的差距。它跟傳統技術債不一樣——技術債會透過越來越慢的 build、越來越糾結的依賴對你發出摩擦訊號；理解債滋生的卻是虛假的信心：測試是綠的、PR 合了、demo 跑得動，直到某天凌晨出事，才發現沒有人真的知道那段程式碼為什麼長那樣。

這不只是直覺，有實驗數據。Anthropic 的研究者在 2026 年 1 月發表了一個隨機對照實驗，找了 52 位工程師學一個新的非同步程式庫。結果：AI 輔助組完成任務的時間跟對照組幾乎一樣，平均只快大概兩分鐘，沒有統計顯著；但在後續的理解測驗上，低了 17 個百分點——百分之五十對百分之六十七。退化最陡的，正是除錯能力，而除錯恰恰是抓 AI 錯誤最需要的人類技能。更值得玩味的是組內差異：把 AI 當家教、問概念的人，測驗拿 65 分以上；把程式碼生成整個外包給 AI 的人，掉到 40 分以下。怎麼用，比用不用更關鍵。

而迴圈把這個問題乘上一個倍數。Osmani 對 loop engineering 的核心警告是：「一個無人看管的迴圈，也是一個無人看管地犯錯的迴圈。」傳統上，資深工程師審查的速度快過初級工程師寫的速度；AI 把這個經濟學翻轉了；迴圈再翻轉一次——現在連下 prompt 的人都不在場了。理解債於是以迴圈的節奏複利成長。

債的盡頭有個更冷的概念，叫 cognitive surrender，認知繳械。它跟「認知卸載」不同——卸載是把工具當工具，判斷還在自己手上；繳械是你從來沒有建構過一個可以拿來對照的平行觀點，AI 的輸出悄悄變成「你的」輸出，而你已經沒有獨立視角去檢查它。當迴圈自己會跑，「不再有意見、照單全收」變得極具誘惑。哲學家 Andy Clark 的區分在這裡很有用：委託產生繳械，協作產生相互放大——你的判斷讓模型的輸出更好，模型的輸出讓你對問題的理解更好。迴圈該被設計成後者的基礎設施，而不是前者的藉口。

把三本帳放在一起，結論不是「別用迴圈」。token 帳單有工程解，三道硬煞車；cron 換皮的批評有架構解，決策外面那一圈才是本體；唯獨理解債沒有自動化解——它只能由一個決定留在現場的人來償還。社群沉澱出的緩解措施都指向同一件事：獨立的 verifier 擋住「agent 自稱完成」、合併進主幹的程式碼仍經人審、工程師留在設計與審查的路徑上。還有一句很值得記住的話：兩個人跑同一個迴圈，可能得到相反的結果——迴圈不知道差在哪，你知道。迴圈放大的是判斷，好的跟壞的都放大。

最後，回到導聆許下的三個問題，做整輯的收束。第一，loop engineering 新在哪？不是排程，是工程師的位置從迴圈裡面移到迴圈外面——你設計的是那個會發現工作、派工、驗證、記住狀態的系統，地基是老的，房子是新的。第二，一個值得信任的迴圈長什麼樣？六塊積木齊全，而且裡面有少而硬、agent 改不到的「不」：唯讀的測試、獨立的 checker、校準好的 human gate，外加三道成本煞車。第三，你的第一個迴圈從哪開始？從你每天真的在浪費時間的地方，從便宜的、只報告的模式，從一段設好上限的腳本開始——先讓它證明看得準，再讓它動手，最後才談無人值守。

Osmani 的收尾值得照念：蓋迴圈吧，但要以「打算繼續當工程師的人」的方式去蓋，而不是只當那個按下開始鍵的人。迴圈會繼續變便宜、變可靠、變無聊——所有成功的基礎設施最後都會變無聊。不會變的是：理解沒有 API，債照樣計息。本期到此，謝謝收聽。
