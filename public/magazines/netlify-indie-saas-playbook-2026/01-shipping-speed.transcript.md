這一集我們先談最痛的點：為什麼很多個人 SaaS 不是做不出功能，而是不敢穩定上線。你可能很熟這種感覺，修一個小 bug，結果擔心把付款或登入又弄壞，所以乾脆拖著不發。拖久了改動越積越多，一次上線風險更高，然後你更不敢發，進入惡性循環。

Netlify 在這一段最有幫助的，是把每次 pull request 都自動變成 Deploy Preview。也就是說，你每次改動都能立刻有一個可分享連結，設計、PM、合作夥伴不用本地拉分支就能直接看。這件事看起來只是方便，其實是把「口頭驗收」改成「有連結、有版本、有時間點」的正式流程。對一人團隊尤其重要，因為未來你回頭看問題時，不會只剩模糊記憶。

再來是 Atomic Deploy 和回滾。Atomic 的意思是新版本全部準備好才一次切換，不會出現 HTML 已更新、資源還沒上去的半套狀態。更關鍵的是，當你發現這次上線有嚴重問題，你可以把舊版重新發佈回去，先止血，再慢慢查。這和凌晨硬修 hotfix 的差別非常大，前者是流程，後者是賭運氣。

你也可以用分支層級的 Split Testing 去做落地頁和 onboarding 實驗。它的好處是不用先自己做完整的實驗平台，就能快速比較兩個版本。不過你要記得它有適用邊界，特別是高度依賴動態回應的頁面要先驗證限制。

最後想給一個很務實的建議：把上線當作固定節奏，而不是情緒決策。每天或每兩天一個 production 窗口，其他時間全部走 preview 驗收。事故出現先回滾，再復盤。只要這套節奏穩了，你的產品迭代速度通常會比以前快，而且壓力更低。
