上一集我們談的是怎麼把 vibe coding 升級成比較像工程的工作流。這一集要聊的，是很多人一開始最不想面對，但最後一定會碰到的問題，也就是宿醉。

所謂宿醉，不是第一天做不出來，而是第一天做得太順。你一路 prompt、一路 accept、一路修，功能也真的跑起來了，畫面也能 demo，甚至第一批用戶都開始用了。問題是，這個順利感很容易讓你以為自己已經掌握了系統。等到幾週後要加新功能、補權限、修資料流、處理安全問題時，才發現你其實只掌握了表面行為，沒有真正掌握系統內部。

這種現象，arXiv 那篇談 vibe coding 的論文用 flow-debt tradeoff 來描述，我覺得非常準。意思是，前面的順滑生成，其實常常是在向未來借理解。你今天省下來的規劃與推敲時間，會在之後以架構漂移、安全缺口、維護負擔的形式回來。

Simon Willison 的說法很適合當這個問題的實務版。他的意思大概是，如果你沒有讀那段 code，也不能向別人解釋它在做什麼，那它就不該算 production-quality 的 AI 輔助工程。這句話聽起來像原則，其實是維護成本的底線。因為真正的可維護性，不是程式能不能跑，而是三週後你還敢不敢改。

安全問題尤其明顯。很多人會把資安想成之後再補的事情，好像產品先上線比較重要。但在 vibe coding 情境下，這種想法特別危險，因為生成工具天生會先幫你完成 happy path，不一定會替你保守處理 secrets、資料暴露與權限邊界。

這點 2026 年 4 月的幾起事件已經講得很清楚。Vercel 的資安公告直接提到，未標記為 sensitive 的 environment variables 有曝光風險，而且他們建議相關用戶優先輪替。重點不只是 Vercel 出了什麼事，而是它提醒所有人，環境變數的標記方式、預設值、密鑰存放策略，本身就是產品安全的一部分。

另一個例子是 Lovable。官方文件對 project visibility 寫得很清楚，public project 代表別人可以在 editor 裡看到 source code、chat history 跟未發佈中的工作內容；後來預設也改成 workspace visibility。這讓我們看到一件很重要的事：在 AI 生產式工具裡，聊天本身就可能是敏感資料。你以為你只是跟工具對話，實際上那段上下文可能包含需求、資料樣本、客戶資訊，甚至一些你沒有意識到的內部邏輯。

最麻煩的地方在於，這種風險不是總會用明顯的 bug 形式出現。更常見的是，你開始不知道該懷疑哪裡。傳統手刻系統有高風險區域，像 auth、payments、uploads、admin panel，但你通常心裡有數。vibe coding 生成出來的系統，最大的風險反而是你以為自己知道邊界在哪，實際上 agent 可能在你沒注意的地方加了一條過寬的權限、多存了一份資料、或把某段 prompt 歷史暴露到不該暴露的地方。

所以我很建議 guardrail 要做，而且要早做。但 guardrail 不代表你得一開始就蓋出企業級資安體系。對獨立開發者來說，最實用的是四件事。第一，把高風險區域列出來，知道哪些檔案碰 auth、billing、secrets、外部寫入。第二，從第一天就把 secret 管理做對，不要寫死 key，也不要以為以後再刪就好。第三，讓 review 跟 blast radius 成比例，高風險 diff 花更多注意力。第四，保留人工兜底能力，也就是出了問題時你還能手動回退、手動修正資料、手動處理核心流程。

如果要把這一集濃縮成一句話，那就是：vibe coding 最大的問題不是速度太快，而是你太晚承認這個速度已經超過你的理解能力。技術債可以慢慢還，信任債通常不能。所以 guardrail 的真正目的，不是拖慢你，而是避免你在最開心的時候，把最難還的債一次借滿。
