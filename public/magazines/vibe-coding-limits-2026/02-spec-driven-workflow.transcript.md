上一集談的是甜蜜區，這一集要往前走一步，聊一件很多人做到一半才發現的事：專案一變大，就不能只靠 prompt 了。

很多人會把這個卡關點理解成模型不夠強。好像前面做小功能都很順，一到稍微複雜的流程、幾個檔案以上的改動、或跨模組調整時，AI 突然變笨了。可是真正的問題通常不是模型智商，而是你的專案已經開始需要共同記憶，而這種共同記憶如果沒有被寫下來，就只能一直靠當下那段對話臨時維持。

你可能很熟這個感覺。這次你把需求講清楚了，下次 context 換了又得重講。這一輪 agent 理解了你的風格，下一輪又自己補了一套新規則。當 repo 還小時，這不是大問題，因為你腦中還裝得下整個系統；但一旦檔案多了、相依關係複雜了，你會發現每次改動都像在重新談判同一套規則。

所以真正需要升級的，不是 prompt 長度，而是把意圖外部化。也就是說，把「你到底想要什麼、哪些地方不能動、哪些地方碰了就要驗證、什麼算完成」寫進 repo 裡。

這就是 spec-driven workflow 的核心。它不一定是很重的 PRD，也不一定是企業那種層層審批文件。對獨立開發者來說，它更像是一份穩定的 feature brief，一份反覆可讀的 markdown，或是一套固定模板。裡面至少要有幾件事：這個功能解哪個痛點，成功跟失敗條件是什麼，不能碰哪些資料或流程，哪些 UI 或 API 行為是硬約束，還有最基本的驗證標準是什麼。

只要這些東西沒被固定下來，agent 每次都會用最 plausible 的答案替你補空格，而不是用你的真正意圖補空格。這也是為什麼很多人感覺 AI 在專案變大後開始漂移。它不是不會做，而是你沒有把應該穩定的東西變成穩定。

Anthropic 的 Claude Code 文件其實已經很明顯指向這條路。它把 subagent 定義成在自己 context window 裡處理特定工作，避免主對話被大量搜尋結果、log 或中間輸出塞爆；hooks 則可以在寫檔或改檔後，自動跑後續命令。這兩個能力合起來，等於在告訴你：與其讓一個 agent 記住全部，不如把不同階段拆成不同角色，並用系統去固定回饋節點。

我自己會建議 solo builder 用很簡單的三步升級。第一步，先把規格寫下來。第二步，把驗證固定下來，例如每次寫完自動跑 lint、型別檢查、單元測試。第三步，把回饋自動化，而不是把生產全部自動化。

這也是為什麼我很喜歡看 Cursor 今年的方向。Bugbot 的重點不是它抓 bug 很酷，而是它代表整個工作流的價值重心，已經慢慢從「生成更多 code」轉到「更快發現哪些 code 不該進 production」。Cursor 的 Automations 也一樣，真正解的是 review、incident、maintenance 這些原本沒有跟上生成速度的環節。

所以如果你問我，從 vibe coding 升級到比較像工程，最先該做什麼，我不會說換模型，也不會說先搞一大堆 infra。我會說，先把三條線拉穩：規格、驗證、角色分工。當你願意承認專案已經超過「靠感覺就能安全前進」的範圍時，AI 才會從玩具變成真正的槓桿。因為從那一刻開始，你用的不再只是生成能力，而是一整套可重放、可驗證、可維護的流程。
