這一集想聊的是，一人用 vibe coding 到底能吃下多大的專案。我的答案先講在前面，你真正吃下的，通常不是大專案，而是低風險專案。

很多人第一次接觸 AI 寫程式，最容易產生一種錯覺，就是把做得快，誤認成做得大。你今天用幾輪對話把一個登入流程、一個儀表板、一個收費頁，甚至一個簡單的管理後台都拉出來了，畫面上看起來像 SaaS，於是很容易覺得自己已經可以一個人做出完整產品。但真正的分界線從來不是你做了幾頁，而是這個東西如果出錯，會傷到誰，會有多難救。

Harvard Gazette 在 2026 年 4 月的訪談裡有一句我很喜歡的話，大意是說，vibe coding 最常優化的是下一個小時能不能更 wow，而不是可靠性、安全性跟可維護性。這句話其實正好講中核心。對獨立開發者來說，AI 放大的不是你突然能處理企業級複雜度，而是你能用更低成本、更短時間把點子做成第一版，然後去驗證市場有沒有反應。

所以一人 vibe coding 的甜蜜區，大概會落在三種東西。第一種是個人工具，或者單人工作流用的工具。像內容整理器、資料轉換器、個人 CRM、內部小面板。這些產品就算出錯，通常傷害範圍也有限，而且很多時候你自己可以人工補救。

第二種是狹窄場景的微型 SaaS。也就是問題邊界非常清楚、功能不一定少，但責任範圍很窄的產品。它也許只處理一種資料型態、只解一條工作流、只幫特定族群省 20 分鐘。這種產品最適合用 AI 快速上線，因為你知道它不負責什麼，也知道出事時誰能手動接手。

第三種，是工程深度沒有那麼高，但分發能力很重要的產品。像 Indie Hackers 今年初訪談的 Connor Burd，經營一個獨立 app studio，每月 recurring revenue 很高，但他真正反覆強調的是 distribution、clean execution 跟 data-driven iteration。這其實很值得記。因為 AI 讓你更快做出東西，不代表 code 本身就變成最稀缺的東西。很多時候，真正決定上限的還是你怎麼找到 audience、怎麼持續優化、怎麼把產品放到對的人面前。

那反過來，什麼時候就不該再純靠 vibe？我的判準也很簡單。第一，開始碰高風險資料時。只要你的產品要處理個資、醫療資料、支付資訊、企業內部文件，風險模型就變了。第二，權限跟狀態一多時。像多租戶系統、approval workflow、複雜角色分權，真正難的不是把畫面做出來，而是每個邊界條件都要對。第三，外部整合一多時。每多一個 webhook、外部 API 或第三方授權流程，就多一個 failure mode。第四，當產品準備長期演進時。因為這時候你最需要的是可維護性，不只是可生成性。

這也是為什麼我會說，一人能吃下的不是大專案，而是低風險專案。只要你還在可控範圍內，AI 是火箭；一旦 blast radius 變大，它就會從加速器變成放大器。

所以重點不是不要用 vibe coding，而是先用它快速找需求、找市場、找出值得做的東西，然後誠實判斷哪些部分還能靠速度，哪些部分已經該升級成工程流程。因為如果你自己都看不懂 AI 生成的關鍵路徑，那不是產品做大了，而是你對產品的控制變小了。這條線一旦搞錯，後面付的代價通常比你一開始省下來的時間大得多。
