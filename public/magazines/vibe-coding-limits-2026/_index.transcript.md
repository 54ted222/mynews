這一集想聊的是，vibe coding 走到 2026，最值得問的問題，已經不是能不能做出產品，而是你到底能用這種方法把產品做到哪一步。

過去一年，很多人第一次接觸 AI 寫程式，感受到的都是速度。你腦中有個想法，丟進去，幾分鐘後就有畫面、有流程、有一個看起來像產品的東西跑出來。這種體感太強，所以大家很容易把它理解成一種單向進步，好像現在只要模型再強一點，獨立開發者就能一路從點子、MVP、收費、成長，甚至走到企業級系統。

但這件事其實沒那麼線性。Harvard Gazette 在 2026 年 4 月初的一篇訪談，講得很準。受訪者 Karen Brennan 一方面說，vibe coding 讓更多人可以把想法快速做出來，它確實改變了 experimentation 的經濟學；但另一方面，她也提醒，這種做法常常優化的是「下一個小時能不能更 wow」，而不是可靠性、安全性與可維護性。這句話幾乎可以當成整期專刊的起點。

所以我們這一期不是要重講 vibe coding 的定義，而是想把它拆成四個更實際的問題。

第一個問題是，對一位會 JavaScript 前後端、想做 SaaS 的工程師來說，什麼樣的專案還在一人 vibe coding 的甜蜜區，什麼樣的專案其實只是看起來做得快，風險卻已經超出控制範圍。因為你真正吃下的，往往不是大專案，而是低風險專案。

第二個問題是，當產品開始變大，你是不是還能只靠 prompt 往前走。很多人做到這裡會卡住，以為要換更強的模型。其實更常見的狀況是，模型沒有突然變笨，而是你的專案已經需要規格、驗證節點、hooks、review 與 agent 分工。也就是說，問題不再是生成能力，而是流程能力。

第三個問題，是我們這期最想提醒你的地方，也就是宿醉。AI 生成程式最可怕的地方，不一定是第一天做不出來，而是第一天做得太順，順到你誤以為自己已經掌握了系統。等到兩週、四週、兩個月後要重構、補權限、換資料流、修安全洞時，才發現自己其實在用未來的理解能力替現在的速度買單。

第四個問題，是市場面。就算你真的能把產品用 AI 很快做出來，客戶也不會因為這件事本身就買單。2026 年真正賣得動的，不是「你看我多快」，而是「這個東西出了問題時誰負責、能不能帶走、能不能接手、風險有沒有被看見」。換句話說，最後的門檻不是 demo，而是信任。

所以這一期的四篇文章，其實是在畫一條分界線。前面談甜蜜區，幫你判斷哪裡能用速度換機會；中間談 spec-driven workflow，告訴你什麼時候必須把規則寫進 repo；接著談 hangover 和 guardrails，提醒你哪些風險不是之後再補就行；最後談 trust 跟 GTM，因為產品做出來不等於能賣，能賣也不等於能讓人放心用。

如果要先給這一期一個總結，那就是這一句：一人 vibe coding 不是不能做大，而是不能一直用同一種方法做大。小專案靠 prompt，成長中的專案靠規格，真正進入商業責任後靠治理。你不需要放棄 AI，但你必須知道什麼時候該放棄「只靠 vibe」。
