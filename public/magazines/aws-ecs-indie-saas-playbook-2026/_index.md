很多個人開發者在選雲端平台時，會先排除 ECS，理由通常是「看起來太企業級」。但實際上，ECS 的價值不是讓你把系統變複雜，而是讓你在**不自己維護控制平面**的前提下，把容器化 SaaS 的關鍵流程標準化：部署、伸縮、排程、權限、觀測、回滾。

這期主題不是要你一開始就把 AWS 所有服務接滿，而是回答一個更務實的問題：**如果你是 1–3 人團隊，使用 ECS 到底能立刻省下哪些營運摩擦？**

我們把問題拆成四段。第一篇先畫「服務地圖」，把 ECS + Fargate + ALB + ECR + CloudWatch + Secrets Manager 這組最小可行棧說清楚。第二篇談交付：怎麼把 rolling update、deployment circuit breaker、blue/green 放到同一條發布路徑裡，避免每次上線都在祈禱。第三篇進入營運現場，聚焦 Service Connect、ECS Exec、Container Insights，處理「壞了怎麼找、怎麼修」。最後一篇只談現金流與擴張曲線：Fargate/Fargate Spot、Auto Scaling、排程任務，哪些情境能省，哪些情境反而會超支。

截至 2026-04，ECS 在官方文件上已把許多過去需要手工拼裝的能力做成預設路徑（例如排程、部署失敗偵測、服務層連線與觀測）。這意味著：個人 SaaS 不一定要先自己寫平台工程，才能進入「穩定交付」階段。本期目標就是幫你判斷，哪些功能該現在用，哪些等有流量再開。
