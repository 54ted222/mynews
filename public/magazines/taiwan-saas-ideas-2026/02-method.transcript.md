歡迎回到 2026 台灣 SaaS 創業現場，這是第二集，我們來聊方法。上一集談的是「要做什麼」，這一集只談「怎麼做」。

先拋一個結論：如果你在 2026 年要蓋一個 SaaS，真正拉開差距的其實不是你挑哪個 AI 工具，而是三件事。第一，你能不能把驗證階段的工具跟產品階段的工具分開來看；第二，你願不願意把所謂 context、也就是給 AI 的背景脈絡，當成一等公民來經營；第三，你知不知道什麼時候該從 vibe coding 畢業。這一集會把這三件事一次講清楚。

先講 vibe coding。這個詞 2025 年還帶點嘲諷意味，就是說你也不看 code，你就憑感覺跟 AI 聊，然後 app 就出來了。那 2026 年，它已經不是一個梗，它是一整個產業類別。我舉幾個數字，Lovable 這家瑞典公司做到每年經常性收入四億美金，每天有 20 萬個新專案被建立出來；Bolt.new 在半年之內衝到 4000 萬年經常性收入；Replit 推出 Agent 3 之後，公司估值從 30 億直接跳到 90 億美金。這些數字合起來只說明一件事，就是非工程師可以在一個下午蓋出一個「看起來像 SaaS」的東西，這件事已經是真的，不是demo。

但是「看起來像」跟「活得下來」是兩回事。2026 年 2 月有一個資安研究員叫 Taimur Khan，他在一個 Lovable 託管的應用裡面找出 16 個漏洞，其中 6 個是 critical 等級，外洩出去的資料有將近一萬九千筆，裡面還包含四千多名 UC Berkeley 跟 UC Davis 學生的帳戶。然後 3 月他又通報一個更嚴重的權限漏洞，叫做 BOLA，簡單講就是任何一個註冊用戶，都可以去讀別人專案的原始碼、資料庫密碼、還有 AI 對話紀錄。這個漏洞被當成重複通報晾了 48 天才修。更可怕的是一份更大規模的掃描，他們掃了一千六百多支用 Lovable 做的 app，發現有將近七成完全把資料庫的 Row Level Security 關掉。

這不是 Lovable 一家的問題喔。Trend Micro 趨勢科技 3 月發表了一份研究，他們掃了五千六百個公開部署的 vibe coding 應用，找出兩千個高危漏洞、四百個外流的 API 金鑰。另外一個很常被引用的數字是，AI 輔助寫出來的 commit，洩漏機密的機率是人類的兩倍，3.2% 對 1.5%。

所以我對 vibe coding 成熟度的判斷是這樣：前面七成神速，最後三成會殺人。你可以用它來證明「這東西有人要買」，但你不能用它來證明「這東西可以上線營運」。這個邊界，就是 2026 版方法學的起點。

有了這個認知，接下來我們來談三層工具分工。我把工具依用途切成三層，每層放棄掉的東西不一樣。

第一層叫驗證層，從零到可以給人看。這一層就是 Lovable、Bolt、v0 這些 AI 全棧生成工具。目標很簡單，48 小時之內，生一個網址出來，丟給 10 個潛在用戶看。這一層要的是速度，不是品質。Lovable 內建 Supabase，所以你需要登入、需要資料庫，它很合適。Bolt 的特色是在瀏覽器裡面跑 Node.js，所以前端彈性最大，但沒有內建資料庫。v0 只做前端，但生出來的 React 跟 Tailwind 程式碼品質乾淨，可以直接貼進 Next.js 專案。台灣團隊在這一層要決定的不是用哪個，而是你願意燒多少 token、多少 credit。因為這幾個工具都有一個共同毛病，就是複雜一點的 prompt 可以一口氣吃掉你一天的額度。

第二層叫原型層，也就是說驗證通過、確定要認真做之後，你要把 Lovable 匯出來的 code 拉到本地，用 Cursor 或者 Windsurf 打開。這一層的工作是什麼？是把 vibe code 的產物變成一個可以做 code review 的東西。Cursor 今年 1 月釋出了 CLI 的 Plan Mode 跟 Cloud Handoff，Plan Mode 是強迫 AI 動手前先問清楚、先產出計畫；Cloud Handoff 是讓你把本機對話推到雲端繼續跑。3 月又上了自架代理跟 JetBrains 整合。這層的精神是「人類還在 loop 裡面」，AI 寫 code、你 review 每一行 diff、你跑測試。

第三層叫產品層，這東西要活兩年。這一層主要是 Claude Code 加上你自己建的基礎設施。Pragmatic Engineer 今年 2 月做了一份九百多人的開發者調查，Claude Code 以 46% 的「最愛」比例排名第一。它的強項不在 UI，是在終端機介面加上一套明確的工作流，包括子代理、平行執行、git worktree 隔離。Claude Code 2 月的 2.1.32 版推出 Agent Teams，多個 AI 代理在同一個 codebase 上平行工作、由一個 team lead 分派任務，有個案例是把原本 4 小時的 feature 壓到 45 分鐘。這一層的精神是「把 AI 當員工」，你寫規格、它寫 PR、你做 code review，跟管一個遠端工程師沒兩樣。

這三層之間要明確交接。我看過最多的翻車案例，就是團隊在 Lovable 上一路做到有付費用戶了，然後某天客戶說我要加第二種使用者角色，結果發現全部要重寫。為什麼？因為第一階段根本沒規劃資料模型，全靠 AI 臨場腦補。所以請記得，驗證層的產物應該被當成一份可以丟掉的簡報，不是 codebase。

接下來講 context engineering，這是 2026 年取代 prompt engineering 的新關鍵字。定義很短：主對話保持小、專案規則明確、記憶分層、工具層刻意設計、驗證步驟不可省。GitHub 上面檔名叫 CLAUDE.md 的 repo，今年第一季已經超過六萬個；企業端的多代理查詢量年增 1445%。

台灣團隊最常犯的錯誤是什麼？是把 CLAUDE.md 寫成 AI 使用手冊，越長越好、越詳細越好。但實際數據打臉。Anthropic 3 月有一份內部研究指出，開發者手寫的 context 檔會讓 AI 任務成功率提升 4%，但是讓 AI 自動生成的 context 檔，成功率反而下降 3%、推論成本還多花兩成以上。所以答案是短、手寫、常更新。我建議最小可行版本只涵蓋四件事，就是技術棧、三到五條不能破的慣例、build 跟 test 指令、還有一個從 code 看不出來的架構決策。超過一百行，通常就是在灌水。

另外一個是 MCP，Model Context Protocol。它是 Anthropic 提出的一個標準協定，讓 AI 代理可以用統一方式接外部工具。這東西的 SDK 月下載量，從 2024 年 11 月的兩百萬，漲到 2026 年 3 月的 9700 萬。GitHub 上面有一萬三千多個 MCP server。對台灣團隊最實際的用法就是把 Supabase、Notion、Linear、或者你公司內部的 API 用 MCP 接上 AI 代理，不用每次都複製貼上 schema。但是 MCP 也不是萬能。Perplexity 的技術長 Denis Yarats 3 月公開說他們改回去用傳統 API 加 CLI，理由是 MCP 吃太多 token、認證太麻煩、AI 的自主性反而下降。所以我的建議是：外部系統用 MCP，內部工具用 CLI 讓 AI 直接呼叫，兩者混用，不要迷信單一協定。

最後一段，什麼時候該從 vibe code 畢業。三個很明確的訊號。

第一個訊號，第一次遇到「改一個地方，壞三個地方」。這代表你的程式碼規模已經超過 AI 一次能看懂的範圍。通常發生在 codebase 超過一萬行、或者有兩個以上的使用者角色。這時候硬留在 Lovable 上，只是在用更多 credit 換更慢的速度而已。

第二個訊號，企業客戶第一次丟來資安問卷。可能是 SOC 2、可能是 ISO 27001、也可能只是一張 50 題的 security questionnaire。問你密碼雜湊用什麼演算法、session 多久會過期、有沒有做 SQL injection 防護。Lovable 跟 Bolt 生出來的 code 你答不出來，因為你根本沒寫過那些地方。Retool 3 月那篇文章把這個現象叫做「企業就緒懸崖」，過不去就卡在中小企業客戶。

第三個訊號，要處理真錢、真的個資、或者真的法律責任。86% 的 AI 生成 code 樣本帶有 XSS 漏洞；今年 1 月有個叫 Moltbook 的 AI 社群 app 上線三天，整個 production 資料庫就外洩，含一百五十萬筆 API token、三萬五千個 email。這個 app 就是用 vibe coding 蓋出來的。一旦你的產品要碰到錢或個資，就該把資安 review 排進流程。最簡單的做法是產品層只用 Claude Code、手寫或 AI 寫的 code 都強制跑一輪 Snyk 或 Semgrep，commit hook 擋金鑰外洩。

最後我想講一句話。從 vibe code 畢業，不代表放棄 AI，而是把 AI 從「幫我寫完這個 app」，降級成「幫我寫這個 function，我來決定它怎麼接進系統」。這兩句話聽起來很像，但是成本差十倍、責任差一百倍。

這是這一集。下一集我們來聊技術棧，Next.js、Supabase、Stripe、還有台灣在地的金流支付怎麼選。我們下次見。
