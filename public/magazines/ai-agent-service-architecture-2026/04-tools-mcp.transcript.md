前兩集，我們聊了 Agent 怎麼想、怎麼記。這一集，我們來聊它怎麼「做事」——也就是工具層。一個 Agent 的推理核心再強，要是沒有手，它就只是個會聊天的盒子而已。而二零二六年談工具層，有一個東西繞不開，叫 MCP，模型情境協定。

我們先從起點講起。工具層最早的形態，叫做函式呼叫。簡單說，就是你用一種結構化的格式，去描述一個函式長什麼樣、接受哪些參數，然後模型在推理的時候自己決定要不要呼叫、要填什麼進去，你的程式碼接住結果再餵回去。這個東西本身很好用。問題不在於它不好用，而在於它「不可移植」。每個工具的接法，都綁死在特定供應商的格式裡，你換一個模型，就得把這些黏合的程式碼全部重寫一遍。

Anthropic 把這個困境，形容成「N 乘 M 的整合問題」。你可以這樣想：假設你有 N 個 AI 應用，要去接 M 個工具或資料源，那你就得寫 N 乘以 M 份各自維護的整合。舉個具體的例子，三個模型要接十個內部系統，就是三十條客製的連線，每一條都要單獨升級、單獨除錯。數量會隨著兩邊相乘膨脹。其實這是微服務時代早就解決過的老問題——當接口沒有標準，連線數量就會爆炸。

那 MCP 的核心主張就很簡單了:在中間插一層標準協定，把 N 乘 M 變成 N 加 M。每個模型只要會講 MCP，每個工具只要實作一個 MCP server，兩邊各自實作一次,就能互通。業界很喜歡用一個比喻，叫「AI 界的 USB-C」——就像 USB-C 用一種接口取代了各家專屬的充電線，MCP 也用一條協定，取代了那一堆專屬的連接方式。

那 MCP 到底怎麼運作呢？它採用三層架構，刻意把職責跟信任邊界切開。你可以想像成三個角色。第一個叫 host，也就是 AI 應用本身，比如說某個桌面助理、某個編輯器，或是你自己建的 Agent 服務，它負責協調模型、管理體驗、執行安全策略。第二個角色叫 client，host 會為每一個要連的 server，開一個專屬的 client，每個 client 維持一條到單一 server 的連線。第三個角色就是 server，它是輕量、聚焦的程序，對外暴露能力。這裡有個關鍵設計：client 跟 server 是一對一對應的，這不是冗餘，而是為了隔離——一個壞掉或惡意的 server，不該污染到另一個。

那 server 對外暴露什麼呢？三種東西，分工很明確。第一種叫 tools，是模型可以呼叫的動作，比如跑查詢、寫紀錄，這是「做事」。第二種叫 resources，是唯讀的資料源，給模型補上下文，這是「讀」。第三種叫 prompts，是預先寫好的模板，把工具跟資源用在最佳路徑上，這是「最佳實務的封裝」。還有一個關鍵的差異是「自動發現」——傳統的 API 你得事先知道有哪些端點，但 MCP 的 client 連上一個 server 之後，可以動態地問它「你有哪些能力」,拿回一份帶說明的清單，模型再自己推理該用哪個。

那它為什麼贏了呢？關鍵其實是競爭對手的背書。MCP 是 Anthropic 在二零二四年十一月推出的，真正讓它變成標準的，是二零二五年三月 OpenAI 正式採用，然後 Google、微軟、AWS 陸續跟進。當買賣雙方都同意用同一條協定，網路效應就鎖死了——工具作者只要寫一次 server，就能被一大堆不同的 AI 應用同時取用。數字也很驚人，到二零二六年三月，官方套件的月下載量破九千七百萬，公開的 server 超過一萬個。

還有一個治理上的轉折也很重要。二零二五年底，Anthropic 把 MCP 捐給了一個中立的基金會，這個基金會掛在 Linux 基金會底下。為什麼這件事重要？因為把一個原本由單一公司主導的協定，交給中立的基金會代管，是它能被當成長期基礎設施來押注的前提。對企業來說，這降低了「供應商哪天突然改規格」的政治風險。

那對我們這一輯的主軸——把 Agent 做成一個獨立、可擴展的服務——二零二六年路線圖裡最有份量的，是「無狀態化」。舊版的 MCP server 必須維持 session 狀態，這在負載平衡之後就是水平擴展的死穴，因為請求必須黏在同一個實例上。新的版本直接把協定核心改成無狀態,結果就是:任何請求可以落在任何一個 server 實例上。換句話說，MCP server 開始長得像一個正規的無狀態微服務，可以直接擺進你既有的基礎設施裡。

不過代價也要誠實講。MCP 把工具標準化成可插拔接口的同時，也把攻擊面標準化了。有一種攻擊叫「工具中毒」，就是污染 server 端的能力描述，誘導 Agent 去呼叫惡意的工具，這本質上是個供應鏈問題。當 Agent 能自動發現、並且信任任意的 server，「它連的到底是誰寫的」就成了核心風險。這部分我們點到為止，工具濫權、護欄跟供應鏈防禦的完整處理，就留到下一集講安全的時候,好好展開。
