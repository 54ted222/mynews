前面幾集，我們讓 Agent 學會了想、學會了記、學會了做事、也學會了跑得久。這一集，我們要面對一個更嚴肅的問題：當這個 Agent 開始有了真實的權限，它要怎麼不闖禍？這就是安全與護欄這一層。

我想先扭轉一個觀念。把一個 Agent 接上工具的那一刻，攻擊面就從「模型會不會說錯話」，變成了「模型會不會幫攻擊者做事」。這是兩件完全不同等級的事。二零二六年的威脅模型，有四個形狀你一定要記住。

第一個，也是最重要的，叫提示注入。而且重點早就從「直接注入」——也就是使用者自己貼惡意指令——移到了「間接注入」。間接注入是什麼意思呢？就是惡意指令藏在 Agent 會讀進情境的任何資料裡：一個網頁、一個工單、一份 PDF、甚至是程式碼裡的一行註解。我講一個二零二六年初的真實案例，是教科書等級的示範。有研究人員在 Google 一個叫 Antigravity 的編碼工具上，發現一個滿分十分、也就是最高危等級的漏洞。攻擊者把指令埋進一個公開程式碼倉庫的原始碼註解裡，開發者把檔案拉進來，Agent 就把攻擊者的註解當成了指令、組裝出攻擊載荷、然後觸發執行——攻擊者全程根本沒碰過使用者的帳號。更致命的是，它還繞過了安全模式，最後把一次單純的檔案搜尋，變成了沙箱逃逸跟遠端程式碼執行。教訓很清楚：你要把所有的檔案內容、工具輸出、第三方的資料,一律當成不可信的輸入。

第二個威脅叫「過度代理」。當一個 Agent 拿到了超出任務所需的權限、工具或自主性，那一次成功的注入，就能被放大成刪資料、發郵件、轉帳。第三個是工具跟協定的濫權，剛才上一集提過的工具中毒，就是把惡意指令埋進工具的描述裡。第四個是資料外洩,被注入的 Agent 把情境裡的密鑰、私有資料，透過工具呼叫送出去，往往還偽裝成正常的開發動作。

那防禦呢？OpenAI 在二零二六年四月的官方指南把話講白了：沒有任何單一的模型層方案，能完全擋住注入。你得靠很多道彼此獨立的防線疊起來，讓繞過其中一道，不等於攻破整個系統。實務上有一張十層的清單，但我想幫你抓出最重要的幾層。

地基那一層，叫「指令層級」。意思是，系統提示的指令，絕對優先於使用者訊息，使用者訊息又優先於工具的輸出；模型永遠不該因為使用者或工具的內容，就去推翻系統層的約束。但這裡要清醒一點：這些都是降低機率的「軟」控制。注入這件事，在二零二六年被越來越多研究者定性為「結構性的、可能永遠無法根除」的缺陷，因為語言模型在架構上，就是分不開「資料」跟「指令」。所以真正撐住信任邊界的，是接下來的硬控制。

如果你問我，只能做一件事的話，做哪件？做這件：把「最小權限」套在 Agent 的身分跟每一個工具上。為什麼？因為注入過濾器是個機率遊戲，永遠有辦法繞過；但權限是確定性的——Agent 沒有的工具，就算被注入成功也用不出來。最小權限的正確顆粒度，不是給 Agent 一個很大的服務帳號，而是針對每個工具、每個資料集、每個動作，分別授權。這就是為什麼它是地基:它把「注入成功率」這個你控制不了的變數，跟「注入成功之後的爆炸半徑」這個你完全控制得了的變數，給解耦了。

數字會說服你。有一份二零二六年的研究量到，百分之七十八的 Agent 部署，在沒有任何確定性政策把關的情況下，執行高風險的工具呼叫——意思是,大多數企業的 Agent，根本沒有任何運行時的護欄，能阻止一個由惡意提示觸發的刪除操作。同一份研究還指出，過度授權的系統,事故率大約是落實最小權限的四點五倍。結論很硬:確定性的政策閘，才是你錢花得最值的地方。

然後是把爆炸半徑關進籠子的兩道防線。一道是沙箱。最小權限決定 Agent「能呼叫什麼」，沙箱決定它「呼叫成功之後能碰到什麼」。二零二六年的沙箱生態，收斂到大概三種隔離方式。一種叫微虛擬機,給每個工作負載一個專屬的作業系統核心,隔離最強。另一種叫 gVisor，它在使用者空間做一個攔截層，把程式發出的系統呼叫接住、轉譯，不用整台虛擬機就達到接近的隔離。共識很直接:純容器不夠,因為它共用主機的核心;你需要微虛擬機或 gVisor，才談得上對抗那些會主動嘗試逃逸的程式碼。

最後一道防線，叫「人在迴路」。高風險、不可逆、涉及金錢或敏感資料的動作，必須有一個人按下確認才放行。這件事在認知上人人都同意，某個調查裡有將近七成的人把它評為必要。可是認知跟落地之間,是一道巨大的鴻溝。

而這道鴻溝，正是這一集想收的尾。有一份報告盤點了三十個消費級的 Agent，只有九個有記載沙箱或虛擬機的隔離。治理面更糟:七成四的組織計畫兩年內導入代理式 AI,但只有兩成一有成熟的治理模型;甚至有三成五的組織坦言,他們「沒辦法關掉」一個失控的 Agent。把這些拼起來，二零二六年的真相不是「我們不知道怎麼防」，而是「我們知道清單,卻沒把它裝上去」。沙箱、最小權限政策閘、人在迴路——這三件 Agent 從原型走向生產時最常被跳過的事，恰好就是信任層的全部重量。我的建議是:先裝政策閘，再裝沙箱，最後補上人工審核。別等到你的一個檔案搜尋功能，變成別人的遠端程式碼執行。
