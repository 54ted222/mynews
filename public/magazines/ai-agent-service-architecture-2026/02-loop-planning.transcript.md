上一集，我們把整輯的地圖攤開來，看了一個 Agent 服務的六層架構。這一集，我們來聊最上面那一層，也是整個 Agent 的心臟：推理核心。具體來說，就是它的迴圈、規劃的方式，還有那場吵了很久、最後終於有結論的爭論——到底該用一個 Agent，還是一群 Agent。

先講一個你一定要先扭轉過來的觀念。如果你把一個 Agent 服務最裡面那層剝開，你不會看到什麼神奇的提示詞，你會看到一個很樸素的迴圈。這個迴圈大概是這樣轉的：感知、規劃、行動、觀察，然後重複。你的程式碼帶著情境去呼叫模型，模型吐出一段思考、後面跟著一個工具呼叫，你的程式碼拿去執行、把結果接回來，下一圈再來。這個模式最早是二零二二年一篇叫 ReAct 的論文定下來的，意思就是「推理加上行動」交錯進行。到今天，幾乎每一個還在運作的自主 Agent，最裡層跑的都是它的後代。

到了二零二六年，這件事被重新取了個名字，叫「迴圈工程」。重點是，當模型本身已經夠強了，工程的勝負手就從「怎麼寫提示詞」，往外移到「怎麼設計這個迴圈」。而且有意思的是，Agent 在正式環境翻車的原因高度集中：沒有設硬性的停止條件，結果跑成無限迴圈；目標定義得太鬆，模型就開始自由發揮；長對話把情境窗撐爆；還有就是沒有設成本上限。你發現了嗎，這些全都不是模型的問題，是迴圈設計的問題。

接著我們聊規劃模式。我喜歡用換檔來比喻——如果迴圈是引擎，規劃模式就是變速箱，而你選錯檔位，它會用很可預測的方式懲罰你，不是延遲變長、就是成本爆掉、要不然就是可靠性崩了。光譜的最左邊是純 ReAct，模型一次只想一步、做一步、看結果，再想下一步。好處是它非常能應付動態的任務，每一步都能根據真實結果修正方向；壞處是每一圈都吃 token，而且是一步接一步序列執行，延遲會一直累積。往右邊走，是先規劃再執行——一口氣把完整計畫先想好，然後照著步驟做。它把昂貴的推理成本前置到規劃階段，之後執行就更快更便宜。有一份對比的數字蠻有說服力的：先規劃再執行的正確率大概九成二，純 ReAct 大概八成五；成本上前者一塊出頭，後者快三塊。所以一個經驗法則是：互動式的東西先用 ReAct；需要嚴格控成本、要可預測的，往先規劃那邊走。實務上常見的，其實是混合——先規劃出骨架，執行的時候保留一點臨場調整的權力。

然後我要講一件可能會顛覆你直覺的事。框架的選擇，它的影響大到可以左右跑分。普林斯頓有一個 Agent 的排行榜量到，同一個模型、同一個任務，光是換一套不同的鷹架,也就是包在模型外面驅動迴圈的那層程式碼，分數就能差到三十個百分點。三十個百分點是什麼概念？這個差距，比很多前沿模型一個世代到下一個世代的進步還要大。換句話說，你看到的那些跑分數字，有一大半其實是鷹架的功勞，不是模型本身。市面上的框架百花齊放，有的把迴圈做成一張有向圖、擅長複雜的有狀態編排；有的用角色化的團隊概念、二十行就能起手；有的刻意不把迴圈藏起來、讓你直接看見並控制它。風格各異，但它們對「迴圈該怎麼編排」的答案，正在悄悄收斂。

最後，就是那場最精彩的爭論：一個 Agent，還是一群 Agent。二零二五年中，這還是火藥味十足的對立。一邊是 Cognition，這家做 AI 軟體工程師的公司，立場最硬，主張只走單線程，理由是一群 Agent 各做各的、彼此看不見對方在幹嘛，協調的成本跟情境的碎裂會把好處吃光。另一邊是 Anthropic，展示了一個主協調者帶著一群子 Agent 平行探索的研究系統，效果比單體高出九成——但代價是大約十五倍的 token。

後來學界給了這場爭論一個理論上的收斂。史丹佛跟 Contextual AI 的一篇論文證明：在固定的推理預算下、而且情境利用完美的時候，單體系統在資訊論上其實更有效率。他們還點破一個關鍵：過去很多單體對多體的比較根本不公平，因為多體偷偷拿到了更多運算資源。有人把這個叫「群集稅」——你以為買的是更聰明，其實多付的是稅。

最妙的是，連最堅定的反對者也轉向了。那家喊著「別建多 Agent」的公司，九個月後居然出貨了一個「Devin 可以管理 Devin」的功能：一個主 session 當協調者、把工作分派給跑在各自隔離環境裡的子 Devin。注意，這不是打臉，這是收斂——它用的正是大家殊途同歸的那個模式：協調者擁有完整脈絡，派出臨時的子 Agent，每個子 Agent 在全新的情境窗裡執行，做完只回傳一段壓縮過的摘要，彼此之間沒有直接通道、沒有共享的可變狀態。

所以二零二六年的工程共識，不是「單體跟多體選一個」，而是：預設用單體，把 token 預算花在更好的迴圈跟情境工程上；只有當任務夠大、方向彼此獨立、答案值得這麼多 token 的時候，才升級成協調者帶子 Agent。那個十五倍的乘數，順利的時候是投資，但要是子 Agent 開始遞迴地一直生出新的子 Agent，它能在幾分鐘內燒光你一整天的預算。這也是為什麼，這一層的決策迴圈跟編排拓撲，是整個 Agent 服務裡，最該先想清楚的一層。
