上一集我們聊了推理核心，也就是 Agent 怎麼想事情。這一集，我們來聊一個跟它同樣關鍵、卻常被低估的東西：記憶跟情境工程。簡單說，就是怎麼讓一個 Agent 跨會話也不會失憶。

先給你一個有點反直覺的觀念。過去兩年業界最大的認知更新，是大家終於承認一件事：情境視窗，不等於記憶。你可能會想，那我把整段歷史、整批文件、整份工具輸出，全部一股腦塞進那個很大的視窗，不就好了嗎？短期看好像很省事，但長期會壞在三件事上：成本、延遲，還有品質。

品質這件事最反直覺，我多講一點。有一份二零二六年的分析，把百分之六十五的企業 Agent 失敗，歸因到「情境漂移」——注意，不是模型能力不夠。意思是說，Agent 在一份已經過時、或者被一堆無關內容稀釋掉的情境上，「正確地」推理，然後產出一個很有自信、但其實是錯的答案。而且最可怕的是，它是靜默失敗的——沒有報錯，只有越走越歪。更糟的是它會複利:每一步大概錯個百分之二，這個誤差一路滾到最後，可以放大成大約四成的失敗率。這個現象常被叫做「情境腐爛」——視窗裡的雜訊越多,模型對關鍵指令的注意力就越被稀釋。

數字也站在「少而精」這一邊。有個長對話的記憶基準測試,把全部歷史塞進視窗的做法，延遲大概十七秒、每次查詢吃掉兩萬六千個 token；而換成一個有檢索的記憶層，延遲砍到零點二秒、每次查詢只要一千八百個 token，等於延遲跟成本都省了大約九成，準確率卻只掉了個位數的百分點。所以結論很清楚：情境管理的好壞，比視窗的大小重要得多。

那記憶具體該怎麼分層呢？實務上分三層最好懂。短期記憶，就是當前視窗裡的工作記憶——系統提示、最近幾輪對話、手上正在處理的工具輸出，它快、但容量有限，一過就忘。中期記憶，是單一會話、或者最近幾次會話的可檢索歷史，你可以想成一個能搜尋的訊息資料庫。長期記憶，才是真正跨會話、跨任務沉澱下來的東西，底層通常落在三種儲存：一種是向量庫，用語意相似度去找；一種是知識圖，記實體跟關係、還有時序的事實；還有一種是結構化的日誌，可以稽核的事件流跟偏好設定。

這裡有一個很經典的心智模型，來自一篇叫 MemGPT 的研究。它的點子是，把作業系統的「記憶體分頁」概念，搬到語言模型上。你可以這樣想像：模型有限的情境視窗，就像電腦的 RAM、也就是記憶體，容量有限但很快；外面那個近乎無限的外部儲存，就像硬碟。然後 Agent 自己會呼叫函式，在這兩者之間搬資料——需要用到的就從硬碟「頁入」到記憶體，暫時用不到的就「頁出」回硬碟。靠著這套機制，給模型一種「擁有無限情境」的錯覺。這正是 MemGPT 這篇論文最原始的貢獻。

落到工程上，常用的手法有幾類。第一類是壓縮跟摘要，當視窗快滿了，就把舊內容換成摘要。第二類是滑動視窗，只保留最近幾輪，再配上階層式的摘要，把更早的內容一層一層濃縮起來。第三類是記憶卸載，把暫時用不到的東西卸到外部檔案，需要的時候再撈回來。檢索這一端也在進化，從最樸素的切塊加相似度檢索，進化到把文件做成樹狀摘要、或者抽成一張實體關係圖再來查。但我要提醒你一件事:這些技巧不是越複雜越好，有評測就發現,高度結構化的系統，建索引跟查詢的延遲可能是輕量方案的好幾個數量級，準確率卻不見得成比例提升。選型要看你的問題長什麼形狀，別為了用圖而用圖。

框架這塊呢，已經分裂成兩種根本不同的哲學。一派的代表是 Mem0，它是你「外掛」上去的記憶層——幾行程式就接上，幾乎不動你原本的架構，最擅長回答「這個使用者偏好什麼」這類問題。另一派的代表是 Letta，也就是 MemGPT 的商業化延續，它走的是「記憶即架構」——把 Agent 本身做成一個有狀態、可以像服務一樣定址的實體，讓它感覺起來就像個持久存在的服務。另外圖派裡也有分工，有的擅長記「在某個時間點、當時什麼是真的」，有的擅長記「我的語料裡，實體之間怎麼連」。

最後我想釐清兩個常被混為一談的東西，順便收個尾。檢索增強生成、也就是 RAG，它是無狀態的檢索——查詢的時候從外部抓相關片段，會話一結束就忘光，它給 Agent 的是「知識的取用」。而記憶,是有狀態的持久化——跨會話存下情境、按需召回，它給 Agent 的是「連續性」。這兩個不一樣。而且我要特別提醒，記憶這條「能寫入」的路徑，正是它的代價所在。能寫，就能被污染——一旦攻擊者能影響長期記憶的內容，之後所有以這段記憶為條件的互動,全都被攻陷了，而且這種污染還能熬過登出、換裝置、會話結束。所以結論是:記憶層絕對不是把資料倒進向量庫就完事，它需要遺忘策略、衝突解析、時間戳治理，還有寫入的防護。把記憶當成一個會腐爛、會被攻擊的活物來工程，才是讓 Agent 跨會話不失憶、又不至於記錯、記髒的關鍵。
