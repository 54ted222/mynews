這一集，我想先把整輯的地圖攤開來給你看。我們不挖細節，就先建立一個大局觀：一個真正能對外服務的 AI Agent，到底是由哪些東西組成的。

開場我先講一個很多團隊都踩過的坑。你把一段寫得很長的系統提示詞、幾個工具的定義、再加一個簡單的迴圈，這三樣東西拼在一起，跑個展示，通了，看起來很厲害。可是一搬到正式環境，它就開始出狀況：聊著聊著就失憶了，跑到第三步崩潰之後又從頭再來，被一封郵件裡的惡意指令騙去刪檔，然後一個月帳單燒掉五位數，卻沒人說得清這些錢花到哪去了。這就是「展示」跟「服務」之間，那一道很真實的工程鴻溝。

我們先看幾個數字，你就懂為什麼成本是第一個要面對的問題。Agent 會把一次使用者的請求，拆成「規劃、選工具、執行、驗證、再規劃」這樣一圈一圈的迴圈，而且每跑一圈，它都得把累積到目前為止的情境，重新塞回模型裡。所以它呼叫模型的次數，大概是一個純聊天機器人的三到十倍。在軟體工程這種比較重的任務上，一個不設限的 Agent 解掉一個問題，成本大概落在五到八塊美金。這意味著，成本不是上線之後才來優化的事，而是你在設計這個迴圈要跑多深、情境要留多長、用哪個模型的時候，就已經被綁死的第一層約束。

那為什麼非得這樣繞圈圈、這麼花錢呢？因為單獨呼叫一次模型，在複雜任務上的準確率會卡在一個天花板。你只能靠把任務拆成很多輪推理、加上工具呼叫，才有機會把可靠度往上推。換句話說，Agent 是用「更多次的模型呼叫」，去換「更高的任務完成率」。而代價就是，你把一個原本確定的 API 呼叫，變成了一個有狀態、會出錯、需要協調的分散式工作流。

接下來是這一集的主菜，我把整個服務拆成六層，你可以想像成從上到下的六個樓層。

第一層，最上面，是推理迴圈跟規劃，這是 Agent 的心臟。這一層最大的爭論，是該用一個 Agent，還是一群 Agent。二零二五年的時候有兩家公司公開對撞，一家喊「別建多 Agent」，另一家發文示範自家的多 Agent 系統贏過單體將近九成。到了二零二六年，大家其實沉澱出一個折衷：用一個握有完整對話脈絡的主協調者，需要的時候才派出幾個臨時的子 Agent，每個子 Agent 自己開一個新的情境窗、做完只回傳一段摘要。

第二層是記憶跟情境。模型本身沒有狀態，跨對話的連續性得靠你外接。這一層沒做好，Agent 就會在長對話裡失憶、自相矛盾，或是把情境塞爆讓成本失控。

第三層是工具層跟 MCP。這個 MCP，是 Anthropic 在二零二四年底提出的模型情境協定，到現在已經是業界的事實標準了——每個月光是開發套件的下載量就破九千七百萬次，被所有主要 AI 廠商支援。它的價值，是把「能力」標準化成一個可以隨插隨用的接口。

第四層是運行時跟持久執行。這一層，是讓 Agent 從「跑一次的腳本」變成「可靠軟體」的關鍵。一個長任務跑到第二十三步崩潰了，不應該從第一步重來。靠的是一種叫做持久執行的技術，它能重放事件歷史，把狀態重建到崩潰前那一步，再接著跑下去。

第五層是安全與護欄。只要 Agent 有了真實的權限，它本身就變成一個攻擊面。這裡有個很經典的說法叫「致命三重奏」——當一個 Agent 同時能存取私有資料、又會接觸到不可信的內容、還能對外通訊，那麼一句注入的提示詞，就足以讓它把機密洩漏出去。

第六層，最底下，是可觀測性跟成本。要把一個機率系統工程化，靠的是把每一次模型呼叫、每一次工具執行，都記成結構化的紀錄。難就難在它的不確定性——傳統除錯假設同樣的輸入會得到同樣的結果，但 Agent 就算把隨機性調到最低，那個刪錯檔的 bug，重跑一次可能就是不會再出現了。

最後我想說，把服務拆成這六層，不只是為了好看，而是為了「分層交付」。你可以先用一個簡單迴圈加最小工具集上線，再一層一層補上記憶、補上持久執行、補上護欄。每一層都對應一個明確的失敗模式，於是「我們的 Agent 還缺什麼」就變成一張可以打勾的清單，而不是一團模糊的焦慮。但要避開兩個陷阱：一個是過度工程，不是每個 Agent 都需要多體協調或重型的持久執行；另一個是只堆上層、忽略下層——拚命調提示詞、加工具，卻沒有存檔、沒有沙箱、沒有追蹤，這種 Agent 在展示裡很亮眼，一上線就會用你查不出原因的方式爛掉。

所以重點是，這六層的順序，本身就是一種優先級的建議。接下來六集，我們就一層一層,把它講清楚。
