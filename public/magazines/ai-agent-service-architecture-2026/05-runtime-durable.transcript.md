前面四集，我們把 Agent 拆成了迴圈、記憶、工具跟 MCP。但這些其實都偷偷假設了一件事：程式會一直跑到結束。這一集，我們要來戳破這個假設。因為在生產環境裡，這個假設根本不成立。

我先給你一個畫面。想像一個研究型的 Agent，要跑三十分鐘，中間要呼叫八十次模型、用十二個工具。結果跑到一半，承載它的工人程序被系統因為記憶體不足給強制砍掉了，或者剛好碰上部署更新、或者雲端那台搶占式的機器被回收了。只要任何一步沒寫進持久層，整條流程就煙消雲散，使用者看到的，就是一個轉了半小時、然後憑空消失的轉圈圈。所以運行時這一層要回答的問題只有一個：怎麼讓 Agent 跑得久，也活得下來。這就是「獨立 Agent 服務」跟「把聊天框接上 API」最赤裸的分界線。

那問題的核心在哪呢？在於「狀態存在哪裡」。多數的展示，會把 Agent 寫成一個簡單的迴圈，而那些迴圈變數——跑到第幾步了、上一步工具回了什麼、累積的中間結果——全都活在程序的記憶體裡。記憶體是不持久的。程序一死，這些狀態就歸零，沒有任何外部系統知道,這個 Agent 曾經跑到第四十七步。

更糟的是重試的問題。當你天真地「重跑」一個失敗的 Agent，它會從第一步開始：重新呼叫所有模型，這就是重新計費；重新執行所有工具——包括那些有副作用的，比如已經寄出去的郵件、已經扣掉的款、已經建好的雲端資源。這種「射後不理」的迴圈，根本沒有「我做到哪了」的概念。

這裡我要劃清一條界線，免得搞混。這一集講的「狀態」，是執行狀態——跑到第幾步、事件日誌、重試計數。它跟前面第三集講的「語意記憶」——使用者偏好、跨會話的長期知識——是兩回事。前者要的是工作流引擎，後者要的是向量庫。這一集只談前者。

那解法是什麼呢？一個叫「持久執行」的東西。它的機制，我用一句話講完：把工作流程的每一步，都當成一筆不可變的事件，依序追加到一份事件日誌裡；崩潰之後呢，不是重跑，而是「重放」這份日誌，把協調狀態還原，再從中斷的地方接著跑。這裡有一個最關鍵的細節：重放的時候，它不會重新執行那些已經做完的動作。已經完成的模型呼叫、資料庫寫入、工具呼叫，結果直接從日誌裡取回來。

我用一個具體場景帶你走一遍。假設一個流程有一百步，跑到第四十七步,承載它的工人 A 崩潰了。這時候另一個工人 B 來接手。B 會做的，是先把前面四十六步的結果，從事件日誌裡「重放」一遍——注意，這個重放是瞬間的，它不會真的再去打那些 API、不會再花一次 token。它只是把狀態快速灌回到第四十六步的樣子，然後從第四十七步，開始真正地繼續跑。前面花掉的 token，一塊都不會重花。是不是很優雅？

這套機制其實不是新東西，這類技術早就存在了。但二零二六年它因為 Agent 而爆紅。有一家專做這件事的公司叫 Temporal，在二零二六年二月以五十億美元的估值，募了三億美元，過去一年營收成長百分之三百八十。背書的名單也很硬，OpenAI、Replit 這些公司都把 Agent 建在它上面。這側面說明了，這一層的需求有多真實。

運行時的另一半，是「服務拓撲」。二零二六年的共識是「有狀態的 Agent，配上無狀態的 API」。意思是,Agent 本身是個有狀態的長命實體——它得記得試過什麼、什麼失敗了、改過哪些檔案；但它對外打的每一個 API 呼叫，幾乎都是無狀態的——身分跟情境隨著每次呼叫帶過去，而不是靠伺服器端的 session。這樣一來，API 層就可以水平擴展，而狀態的重擔，就集中在工作流引擎跟記憶層。

不過，該潑的冷水也要潑。持久執行有它真實的稅。它的重放模型，要求工作流的程式碼必須是「確定性」的——你不能直接取系統時間、不能直接取亂數、不能在工作流裡直接打 API，這些都得隔離出去。為什麼？因為重放的時候，如果這些值變了，程式就會走出不一樣的路徑，然後出錯。這套心智模型對團隊是有門檻的，硬套在小專案上就是殺雞用牛刀。那家公司自己的講法叫「複雜度懸崖」——當流程長到一定程度，框架會崩，這時候持久執行才變成必需；反過來說，在懸崖之前，它不是必需。

替代方案也不只一種。有些走「存檔點」路線，每個節點完成後把狀態寫進資料庫，適合天生是圖形狀、單體部署的工作流。但這裡有個常見的誤解要小心：有一篇被廣傳的文章直言「存檔點不等於持久執行」——存檔點只給你一個存檔的點，但是誰來偵測失敗、誰來觸發恢復、怎麼防止兩個程序同時恢復同一個任務造成重複執行，這些全都甩回給你自己處理。

所以選型真正的問題，不是「要不要持久化狀態」，而是「我要不要連失敗偵測、分散式協調、工人接管,也一起買」。短流程、單機、能接受偶爾重跑的 Agent，自己刻一個冪等加資料庫存檔，往往就夠了，沒必要扛一整套重型平台。但跨服務、長命、副作用很貴、又要合規稽核的 Agent，自建這套的成本，通常遠超過託管平台的學習曲線。底線就是：運行時這層決定的，不是 Agent 聰不聰明，而是它是不是一個能在生產活下來的服務。先量你的流程會不會跨過那道複雜度懸崖，再決定付不付這個稅——而不是反過來，先選了平台，再去找需要它的理由。
