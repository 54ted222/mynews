前面六集，我們把 Agent 從推理迴圈，一路蓋到了安全護欄。但是蓋好，不等於營運得起來。這最後一集，我們要回答兩個工程經理在開會時真正會問的問題：第一，你怎麼知道這隻 Agent 做得好不好？第二，它一個月燒掉的錢，到底划不划算？

而這兩題都沒有單一指標能回答，原因就在於：Agent 是一個機率系統。同一個輸入，跑兩次，可能走兩條完全不同的路、給你兩種不一樣的答案。把這樣一個機率系統，當成確定性的軟體來監控、來計費，是二零二六年仍然在量產的事故來源。

我們先聊可觀測性。傳統的應用監控工具，它的世界觀是這樣的：請求進來、走幾個固定的步驟、回個成功或失敗，慢了就看是哪一段卡住。但 Agent 不長這樣。一個任務,可能呼叫模型三到十次,中間穿插工具執行、規劃、重試、驗證，而且路徑是模型當下臨時決定的。所以出事的時候，你要回答的不是「哪一段慢」，而是「它當初到底為什麼決定那樣做」。一行一行流過去的日誌，答不了這個問題。

你需要的東西，叫做「追蹤」。我幫你建立一個畫面：你把整個任務，攤成一棵樹。樹上的每一個節點,代表一個工作單位——可能是一次模型呼叫,也可能是一次工具執行。每個節點都帶著它的資訊:用了哪個模型、花了多少 token、輸入輸出是什麼、耗時多久。而節點之間的父子關係，剛好就保留了 Agent 一步步決策的脈絡。這是一門有自己資料模型的學科——它的單位是模型呼叫跟工具呼叫，不是傳統的網路請求;它的品質訊號是評測分數，不是錯誤率;它的成本模型是按 token、按使用者去歸因，不是看 CPU 用了幾秒。好消息是,現在有一套通用的標準慣例,正在把這套格式統一起來,雖然到二零二六年中,很多欄位都還標著「實驗性」。

不過追蹤的量會爆掉，因為 Agent 比聊天多三到十倍的呼叫，每個任務動輒五萬到五十萬個 token，全部留存又貴又吵。所以共識的做法是「尾端取樣」——先把整筆追蹤收完，等它跑完之後再決定要不要留，把出錯的、變慢的、成本異常的優先保下來，正常流量抽樣就好。

再來聊評測。可靠的 Agent 評測，要疊三層。第一層，對那些離散的步驟做單元測試——能用程式判對錯的，就別交給模型,比如工具有沒有被正確呼叫、輸出符不符合格式,這些寫成斷言最便宜也最穩。第二層，對那些沒有標準答案的開放式輸出，比如摘要好不好、回覆切不切題，用「一個模型當評審」去打分，固定成一組黃金測試集，每次改動都重跑，把原本悄無聲息的退化，變成看得見的分數下滑。第三層,把同一批評測掛到生產上,對抽樣的真實流量即時打分。

但這裡有個大坑，必須講清楚。第二、三層都壓在「用模型當評審」這件事上，而這個評審本身，就是二零二六年最被質疑的環節。有研究發現，用兩個評審跑同一批任務、各重複五十次，結果一致性低到作者把論文標題訂成「擲硬幣的評審」。前沿模型在偏誤測試上的錯誤率甚至超過五成。已知的系統性偏誤包括:偏好比較長的、看起來權威的、格式漂亮的答案，甚至會偏心跟自己風格相近的輸出。光是改個格式、換個說法，評審的判斷就可能崩掉。所以結論不是棄用評審，而是把它當成一個「需要被校準的量測儀器」，而不是真理——用人工標註的小集合,去驗證它跟人類的一致率,固定它的設定,把評審自己也納入監控。

最後，我們聊成本，而這也是整輯最後一個、我覺得最深刻的觀點。先看數字:Agent 一個任務普遍五到八塊美金。最大的省錢槓桿叫「提示快取」——對那些重複的前綴，比如固定的系統提示、工具定義，打個快取，命中的時候大概能省九成。再疊上把簡單步驟交給小模型的路由，整體省個七到九成是合理的。

但這裡有個比省錢更重要的觀點，來自一個叫「可靠性稅」的框架。意思是,Agent 把機率的不確定性,灌進了原本確定性的軟體棧,於是你得多付一筆「不可靠稅」——錯誤的圍堵、驗證的關卡、重試的抑制、人為的審核點。這些東西不產生任何新功能，它們只是把失敗的機率壓住。而關鍵的差別在於:省 token 這件事，有個完工日——快取做完、路由調好，就大致到頂了，它是一次性的工程。可是可靠性工程，沒有完工日，因為它對抗的，是模型本質上的隨機性。而且失敗會複利:單一步驟九成五的成功率,十步串起來,就只剩下大約六成。

這也解釋了，為什麼便宜的 Agent 常常不可靠，而可靠的 Agent 常常不便宜。要把可靠性往上拉，目前主流的手段就是「讓它想久一點」——多想幾輪、生成多個答案再挑最好的，但這些直接換成更多的 token 跟呼叫。不過二零二六年的研究也潑了冷水：這種「想久一點」是有天花板的，加太多互動回合，在某些任務上反而會傷害表現。所以營運層真正的工作，是用可觀測性跟評測，去找出「在哪些任務、加多少算力，可靠性的邊際報酬還划得來」，而不是無腦地把預算往推理時間倒。

好，到這裡，我想把這整輯的六層，幫你串回來收個尾。推理核心，讓 Agent 會想;記憶跟工具，讓它有料;運行時，讓它能跑久;安全護欄，讓它不闖禍;而最後這一層,可觀測性、評測跟成本，是讓前面五層的不確定性，變成可以量測、可以定價、可以治理的數字。少了它，你的 Agent 服務,就只是一個你看不見、也算不清帳的機率黑箱。

所以回到我們第一集講的那句話:在正式環境裡，Agent 不是一段提示詞，而是一套分散式系統。走完這六層你會發現，做一個能用的 Agent 服務，靠的真的不是把提示詞調得多漂亮，而是把這六層的工程紀律,一層一層補齊。模型會越來越強、token 會越來越便宜，但這套組裝的功夫，才是會留下來的東西。這一輯就到這裡，謝謝你聽完，我們下次再見。
