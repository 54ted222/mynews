過去兩年，RAG 最常見的誤區是把它當成「把文件塞進向量資料庫就會變好用」的魔法。真正上線後大家才發現：回覆看起來流暢，不代表檢索可靠；偶爾答對，不代表系統可維運；demo 能跑，不代表你能承擔真實流量、延遲、成本與風險。

可靠的 RAG，本質上是**資訊檢索系統 + 生成系統 + 產品治理**的組合工程。它的核心不是「模型多大」，而是你是否把每一層的失敗模式拆開管理：資料如何切塊、如何召回、如何重排、如何驗證、如何回滾、如何追責。這也是為什麼同樣用同一個 LLM，產品表現會差非常多。

這期我們採用「Node.js + PostgreSQL + pgvector」作為討論底座，不是因為這套永遠最強，而是它在可控性、成本與遷移彈性之間，對獨立開發者與小團隊通常最平衡。尤其 pgvector 已經把 halfvec、HNSW、IVFFlat 等必要能力放進 Postgres 世界，代表你可以在單一資料平面裡把交易資料、權限與檢索索引一起治理。

但只靠向量檢索通常不夠。Anthropic 在 Contextual Retrieval 的工程文章中，直接展示了 BM25 + embedding + reranking 的改善幅度，並提醒「知識庫夠小時，整包放進 prompt 反而更簡單」。這句話非常關鍵：可靠不是「永遠更複雜」，而是能根據規模切換策略。

所以這一期不講炫技，而是回答四個會決定你系統壽命的問題：

1. **邊界怎麼畫**：什麼時候該用長上下文、什麼時候該上 RAG、什麼時候該混合。
2. **檢索怎麼做穩**：chunk、hybrid search、reranker 到底如何串成可調參的管線。
3. **品質如何量化**：沒有離線評測就沒有可靠性，只有錯覺。
4. **上線後怎麼不失控**：成本、延遲、權限、資料生命週期如何治理。

你會看到一個很務實的結論：RAG 的護城河不是「你有沒有向量資料庫」，而是你有沒有把檢索與評測當成產品核心能力。若沒有，系統會在流量上來時崩；若有，RAG 才能從功能變成資產。
