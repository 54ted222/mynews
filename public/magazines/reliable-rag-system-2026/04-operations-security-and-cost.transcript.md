來到最後一集，我們談最常在專案後期才被重視、但其實一開始就該設計的部分：營運治理。簡單說，就是成本、權限、版本與事故回應。

先從成本開始。RAG 系統的成本不是只有模型 token，還包含檢索、重排、快取、索引重建和資料儲存。很多團隊前期靠手感估算，後面流量上來就發現每個查詢都走最昂貴路徑，結果是功能越成功、虧損越快。

解法不是盲目砍品質，而是建立分級路由。低風險問題用較便宜路徑，高風險問題才升級高成本路徑。再加上上下文預算、快取和預算熔斷，讓系統在尖峰時還能可控運作。你要讓成本跟價值對齊，而不是讓所有查詢享受同樣豪華待遇。

第二塊是權限。只要進入多租戶場景，權限不能靠提示詞說「請不要回答別人的資料」。你必須在檢索前就把資料面隔離做好，像是 tenant 篩選、角色 ACL、敏感欄位遮罩與稽核日誌。沒有這層保護，任何一次命中錯誤都可能變成資安事件。

第三塊是版本治理。這一點在事故發生時超級關鍵。你必須能追溯某次回答到底用了哪個文件版本、哪組索引、哪個模型與哪版 prompt。沒有這些追蹤資訊，你就無法復盤，也很難對內對外負責。

我建議把 index version、document version、model version、prompt hash 和 trace id 全部寫進推論紀錄。平常看起來很囉嗦，但只要遇到客訴、法務或合規查核，這些欄位會立刻變成你的救命索。

第四塊是 incident response，也就是事故劇本。RAG 的事故常見不是系統掛掉，而是系統還在線、但開始答錯，甚至答到不該答的資料。你要有明確的分級：敏感資料外洩是 P1，要立即封鎖相關索引；高風險領域錯答是 P2，要切人工審核；一般品質退化是 P3，要回退策略並重跑回歸測試。

這裡最重要的是「一鍵回退」能力。你應該能快速切回上一版索引、上一版重排、上一版提示策略。沒有回退按鈕，再完整的監控也只是告訴你：你壞掉了。

最後，為什麼我們在整期一直提 Postgres 與 pgvector？不是因為它一定是最強，而是它能把檢索、權限、審計、備份放在同一治理平面，讓小團隊更容易把制度建立起來。等規模到了，再拆分成更多專用服務也不遲。

我們用一句話收尾：可靠 RAG 的終局不是「更會回答」，而是「在真實世界裡可持續運作」。當你把成本控住、權限守住、版本追得住、事故處理得住，RAG 才不只是炫技，而是可商業化、可長期累積的核心能力。
