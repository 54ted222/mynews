# 合規雙牆：電子發票與 2025 個資法修法

## TL;DR

- **電子發票牆**：個人戶月營業額 8 萬以下不必開、8–20 萬要稅籍登記但可免開、超過 20 萬一律開統一發票；公司戶從第一塊錢就要。B2C 開立後 **48 小時內**上傳財政部、B2B **7 天內**——這個時限是平台還是你 cron job 在扛，必須一開始就決定。
- **個資法牆**：2025/11/11 總統公布修法、個保會（PDPC[^pdpc]）成立、施行細則於 **2026/1/22 預告**並設 **6 年過渡期**。事故通報新制砍掉「違反本法、查證後」的舊門檻——只要「知悉」遭竊／外洩就要在 **72 小時內**通報主管機關與當事人，不准等查清楚。
- **indie 落地策略**：金流平台代開電子發票（綠界[^ecpay]／ezPay[^ezpay]）是 indie 唯一合理路徑，省掉自接 MIG 4.0[^mig40]、稅籍、字軌號碼[^trackcode]整套地獄；個資面則要一開始就把 minimization、加密、log 保留期、事故 SOP 寫進產品文件——修法把責任直接綁在「資料控制者」身上，等出事再補來不及。

## 第一道牆：電子發票，誰開、什麼時候開、誰幫你開

### 個人戶 vs 公司戶：8 萬／20 萬／公司，三條線

很多 indie 一開始就卡在「我還沒成立公司，能不能收錢開發票」。台灣的稅籍邏輯是這樣的：

- **月營業額 < 8 萬**（銷售勞務 < 4 萬）：起徵點以下，免辦稅籍登記、免開發票，個人綜所稅報執行業務所得就好。MVP 階段大部分 indie 在這條線下。
- **月營業額 8–20 萬**：要辦稅籍登記，可申請「免用統一發票」，國稅局核定營業稅率 1%、按季開單繳納。對只賣訂閱的 SaaS 是過渡期最舒服的狀態。
- **月營業額 ≥ 20 萬**：強制開立統一發票、營業稅率 5%、可申報進項稅額扣抵。
- **公司戶（有限公司／股份有限公司）**：不論營收多少，從第一張收據就要開統一發票。設公司那一刻起，電子發票串接就必修。

選擇成立公司的 indie 多半是因為要簽 B2B 合約、申請雲市集補助、或開發票給有統編的客戶——那一刻起，電子發票就是上線的硬阻擋。

### B2C 48 小時、B2B 7 天：這條時限改變架構

財政部「電子發票實施作業要點」對上傳時限有明確規定：**B2C 發票開立後 48 小時內、B2B 7 天內**要上傳到電子發票整合服務平台（截至 2026-04）。聽起來寬鬆，但對 indie 是個架構決策：

- **金流平台代開**（綠界 ECPay 收款 + ECPay 電子發票，或 ezPay 金流 + ezPay 電子發票）：上傳時限是平台幫你扛的，你只要在 webhook 收到 `Invoice/Issue` 成功回應、把發票號碼存資料庫即可。
- **自接財政部 MIG 4.0**：要自己申請字軌號碼、自己寫上傳 cron、自己處理重送與失敗重試、自己對帳。對 indie 等於再多一個產品。

實務上，indie 階段幾乎沒有理由自接 MIG 4.0。發票送平台、平台送財政部，是時間換成本最划算的選擇。

### 綠界 vs ezPay：費率與 SDK 取捨

兩家都是台灣常用的電子發票平台。截至 2026-04：

- **綠界 ECPay**：2026/01 起新戶**首年 0 元、首年最多 20,000 張**；超過張數或第二年起依方案計費。SDK 完整，PHP / Java / .NET 都有，2025 年起補上 Node.js 與 Python sample，並推出「AI Skill 套件」——把開立、作廢、查詢包成 LLM 可呼叫的 tool definition，方便接到 agent 流程。
- **ezPay（藍新系）**：費率類似但首年免費的優惠較窄，文件偏 RESTful、HTTP form-encoded；好處是與藍新金流綁得緊，如果你金流已經選藍新就直接用。

如果你還沒決定金流，把這道牆和 Vol.2 那期談的金流路徑放在一起看：**綠界＋綠界發票** 是 indie 在台灣最少摩擦的組合，因為 webhook 一條鏈、對帳一個後台。

### 開錯與作廢：成本不對稱

電子發票開錯不是改個欄位就好。發票一旦上傳到財政部就視為開立，要修正只能：

1. **同月**：作廢原發票、重開一張新號碼。
2. **跨月**：開立折讓單沖抵，原發票不能作廢。

API 上 `Invoice/Invalid` 與 `Invoice/Allowance` 是兩支不同的端點，回傳碼與錯誤情境也不一樣。indie 的最常踩雷：**測試環境直接打到正式環境字軌**——財政部那頭真的會收下，你一個下午就用掉 50 張正式字軌。記得永遠先用 stage、stage 用 `MerchantID` 切，不要用 if/else 切環境。

## 第二道牆：2025 個資法修法，indie 的真實風險面

### 修法時程與 PDPC 成立

- **2025/10/17**：立法院三讀通過個資法部分條文修正案。
- **2025/11/11**：總統公布。
- **2026/1/22**：個人資料保護委員會（PDPC）籌備處預告事故通報辦法草案。
- **6 年過渡期**：PDPC 將在過渡期內陸續發布施行細則、行業指引、事故通報實施日。完整制度上線預估落在 2030–2031 年間。

但「過渡期」不代表 indie 可以擺爛——**事故通報義務在母法已生效**，等 PDPC 發布細則只是把「怎麼通報、通報誰」具體化。

### 事故通報：72 小時、不能等查清楚

舊法的事故通報觸發條件是「違反本法、查證後通知當事人」——很多企業躲在「我還在查」拖延。新法把這層門檻拿掉：**只要「知悉」資料遭竊取、竄改、毀損、滅失或洩漏，就要立即採取應變措施並通報**。

依 PDPC 草案，**72 小時內**要完成：

1. 通報主管機關（PDPC）。
2. 通知受影響當事人。
3. 留存事故紀錄供日後稽核。

對 indie 的衝擊：你不能再用「等我查清楚再說」的心態。產品出事的那一刻，**時鐘從你 on-call 收到 alert 就開始走**。

### 罰則結構：從萬元到行政限期

修法後罰則結構大致是：

- **未盡安全維護義務**：限期改正，逾期可連續處罰，金額由 PDPC 依規模裁量（草案下限拉高至數十萬起跳）。
- **未通報、未通知當事人**：另一條獨立罰則，與安維義務疊加。
- **重大事故、跨境傳輸違規**：最重可達數百萬至千萬等級，並可命令暫停或限制處理個資。

對 indie 而言，金額不是最痛的——**「命令暫停或限制處理個資」這條才會直接讓產品下線**。一個訂閱 SaaS 被命令暫停處理會員資料，等於業務歸零。

### indie 怎麼設計才不踩雷

修法的精神是**「資料 minimization + 可問責性」**。把它翻成 indie 開發動作：

- **能不收就不收**：phone, address, ID 這類欄位在註冊頁要說得出商業必要性，否則就拿掉。
- **資料分層加密**：DB column-level encryption[^cle] 不是 over-engineering，是 PDPC 行業指引預期的「適當技術措施」。
- **log 保留期定義清楚**：事故通報要交得出 log，但 log 本身又是個資——保留太久變新風險。建議 30–90 天滾動。
- **第三方 vendor 清單**：Postmark、Sentry、Mixpanel、Cloudflare 全部都是「資料處理者」，要列清單、有 DPA[^dpa]。
- **on-call SOP 內含通報流程**：alert → 90 分鐘內初判是否為個資事故 → 若是，啟動 72 小時時鐘。

## indie 開發者合規 checklist

上線前一次過：

- [ ] 確定營業形態（個人戶 / 行號 / 公司）並完成稅籍登記。
- [ ] 金流平台選定，並開通該平台的電子發票服務（不要分兩家）。
- [ ] webhook 收到金流成功事件後，**同一筆 transaction 內**呼叫發票開立 API。
- [ ] stage / production 字軌與 MerchantID 物理隔離，不用 if/else。
- [ ] 發票作廢、折讓、查詢三條 API 都寫過至少一次的 e2e 測試。
- [ ] 隱私權政策更新到符合 2025 新法用語（「個人資料保護委員會」而非「目的事業主管機關」）。
- [ ] 註冊欄位逐一檢視、寫商業必要性註記。
- [ ] DB 敏感欄位 column-level encryption。
- [ ] 事故通報 SOP 文件化，oncall runbook 第一頁就是它。
- [ ] 第三方 vendor 清單與 DPA 收齊。
- [ ] 每季 review 一次 log 保留期、未使用欄位、僵屍帳號清理。

把這張清單跑完，這兩道牆就過了。剩下的就是去把產品做好——下一篇談 LINE OA、TapPay、Fugle 這些「只有台灣才有」的整合層。

[^pdpc]: PDPC（個人資料保護委員會）為台灣依 2025 年修法新設的個資專責主管機關，對應歐盟 GDPR 體系下的 DPA 角色，負責解釋個資法、調查事故與裁罰，籌備處於 2025 年掛牌、預計六年內完成全面接管。

[^ecpay]: ECPay 綠界科技為台灣中小型網路商家最常用的金流與電子發票服務商之一，提供信用卡、ATM、超商代收與電子發票一條龍 SDK，是 indie 開發者最常見的金流選擇。

[^ezpay]: ezPay 為藍新科技（NewebPay）旗下的金流與電子發票品牌，與綠界齊名，定位類似但介接文件偏 RESTful、與藍新金流綁得緊。

[^mig40]: MIG 4.0（Message Implementation Guideline 4.0）為財政部規範的電子發票交換訊息標準，所有自接財政部上傳的業者都要遵循該規格送出與接收 XML 訊息，技術門檻較高。

[^trackcode]: 字軌號碼是統一發票的開頭兩位英文字母編碼，由國稅局每兩個月配發給開立業者，用以區分發票歸屬與防偽；自接電子發票業者必須自行管理字軌與流水號的發放與作廢。

[^cle]: column-level encryption（欄位級加密）指在資料庫單一欄位層級套用加密金鑰，使機敏資料即使整張資料表外洩也無法直接讀取，是個資與金融類產品常用的「適當技術措施」之一。

[^dpa]: DPA（Data Processing Agreement，資料處理契約）為資料控制者與第三方資料處理者之間簽訂、規範資料用途與責任歸屬的合約文件，源自歐盟 GDPR 規範，台灣 2025 修法後也成為實務必備。
