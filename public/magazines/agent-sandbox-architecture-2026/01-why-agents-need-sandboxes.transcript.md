這一集，我們要先回答最根本的一個問題：為什麼 agent 非要沙箱不可？或者換個問法——到了二零二六年，「容器不是沙箱」這句話是怎麼從一句口號，變成大家都認的共識的。

我們先把一件事講清楚：危險的其實不是模型本身。你把一個聊天機器人跟一個 agent 放在一起比，威脅模型完全是兩回事。聊天機器人最壞的情況，是它輸出一段有害的文字，講錯話而已。但 agent 的最壞情況是什麼？是它寫了一段你從來沒看過的程式碼，然後在一個能讀你檔案、能連你內網、能呼叫你 API 的環境裡，把這段程式碼直接跑掉。你聽得出來，這個差別不是程度上的差別，是性質上的差別。

有一位很有名的安全研究者 Simon Willison，把這個結構叫做「致命三重奏」。意思是說，當一個系統同時具備三件事——能存取私有資料、會接觸到不可信的內容、還有一條能對外傳輸的管道——那攻擊者只要能在 agent 會讀到的任何一個地方寫進東西，一封郵件、一個網頁、一份被檢索的文件、甚至一個 issue 留言，他就能改寫這個 agent 的行為。這裡最要命的痛點是：大型語言模型，它本質上分不清「指令」跟「資料」。對人來說，郵件正文當然是資料嘛；但對模型來說，那段文字跟系統提示一樣，都只是 token，它沒辦法天生地把這條界線守死。而且任何過濾器也守不死，因為它頂多擋住已知的攻擊樣式，擋不住攻擊者用無限多種方式重新換句話說。

所以這就是為什麼，「再訓練一個更聽話的模型」是解決不了問題的。問題的根源在系統架構，不在模型對齊。只要 agent 又讀得到不可信輸入、又能執行程式碼、又有出口，它就是一台等著被別人遠端編程的機器。沙箱的角色，不是讓 agent 變乖，而是承認它遲早會不乖，然後替「不乖之後」設一道有上限的損失。

光講道理可能太抽象，我們看幾個二零二六上半年真的被打穿的案例。

第一個是分水嶺，叫 EchoLeak。它是微軟 365 Copilot 上的一個零點擊漏洞。零點擊的意思是，攻擊者只要寄一封郵件給你，你完全不用點任何東西，Copilot 在背景處理你收件匣的時候,就會被郵件裡藏的隱形指令接管,然後把你能存取的 OneDrive、SharePoint、Teams 內容偷偷送出去。它技術上串了好幾層繞過，但重點是，它是第一個在正式的 LLM 產品裡，真的被武器化、真的造成資料外洩的提示注入。它證明了剛剛講的「致命三重奏」不是黑板上的理論，是能打穿企業級產品的真實攻擊鏈。

第二個案例叫 n8n。n8n 是一個自動化平台,常常握著通往內部 API、資料庫、各種第三方服務的憑證金庫。它的漏洞展示了另一個維度的可怕:當一個 agent 或自動化平台,坐在你基礎設施的樞紐位置上,一次沙箱逃逸的爆炸半徑有多大。一旦逃出來,攻擊者能解密所有憑證、執行系統指令,把這台機器當成打進整個內網的跳板。而引爆它,只需要一個能編輯工作流的內部帳號。

第三個案例,我覺得最直指這一期的主題,代號叫 Claudy Day。研究者在 claude.ai 上串了三個漏洞,其中最關鍵的一個是——濫用沙箱的出口白名單。你想喔,Claude 的程式碼執行沙箱,本來就限制對外連線的,但它允許連到 Anthropic 自己的 API。攻擊者就把自己控制的金鑰藏進隱形提示,叫 Claude 把對話歷史裡的敏感資料寫成檔案,再透過那個被允許的 API 上傳到攻擊者的帳號。換句話說,出口白名單上那唯一一個「合法」的網域,就變成了資料外洩的管道。這個案例打臉了一種常見的安心感:我有沙箱、我有限制出口,並不等於安全。只要出口不是零,那條被允許的路,就可能被當成外洩通道。

把這三個案例放在一起看,攻擊鏈的形狀就清楚了:先汙染輸入,再取得執行,然後沿著被允許的出口把資料外洩或橫向移動。而沙箱要守的,正是中段到尾段這一塊。

那回到一開始的問題:「容器不是沙箱」這個共識,是怎麼來的?道理其實不複雜。容器,像 Docker 那種,本質上是打包跟資源控制的機制,它只是限制一個程式看得到什麼、用得到多少資源,但同一台主機上所有容器,是共用同一顆核心的。當你的 agent 能自己寫程式、裝套件、操作底層的時候,它離「一次系統呼叫就打穿主機核心」其實只有一步之遙。而且這不只是直覺,二零二六年也有了量化證據:有一篇論文做了個測試,發現前沿的模型經常能識別並利用漏洞、逃出容器,而嘗試一次逃逸的成本,大概只要一美元。當逃逸的代價是一杯咖啡的零頭,把不可信程式碼丟進共享核心的容器,就不再是一個能長期成立的賭注了。

不過,我得補一個很重要的反方觀點:「容器不是沙箱」,並不等於「換成 microVM 就高枕無憂」。一來,就算是硬體虛擬化,也有自己的攻擊面,只是小很多;二來,更根本的是,提示注入這個源頭本身防不死。你換再強的隔離,也擋不住 Claudy Day 那種「沿著合法出口外洩」的攻擊,因為它根本沒逃逸,它走的是正門。所以真正成熟的設計假設,不是去找一個完美的沙箱,而是假設 agent 一定會被接管,再用分層防禦把損失壓到可接受。

所以這一集的重點是:沙箱之所以在二零二六變成 agent 堆疊裡最貴的一層,不是因為它技術上最難,而是因為它承擔了最重的責任——它是在「模型一定會犯錯、一定會被注入」這個前提下,唯一能把企業損失劃出邊界的那道牆。那既然要這道牆,下一集我們就來看,市面上到底有哪幾種材料能砌這道牆,它們各自的取捨又是什麼。
