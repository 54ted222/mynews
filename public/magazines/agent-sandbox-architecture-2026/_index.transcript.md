這一期我們想聊一個聽起來很技術、但其實正在決定一堆 AI 產品能不能上線的東西，叫做 agent 沙箱。先講一句結論：到了二零二六年中，做 AI agent 最難的地方，已經不是模型本身了。模型之間的差距其實在縮小，真正卡住產品化的，是另一個問題——當你的 agent 會自己寫程式、自己執行、又能自己連網，你到底要把它關在哪裡？有一份產業分析講得很直白，說沙箱是整個 agent 堆疊裡最被搶的一塊地，因為它同時牽動三件事：安全邊界、冷啟動的延遲，還有你每個月的帳單。麻煩就在於，這三件事很難同時拿到，你常常是顧了這個就傷了那個。

可能有人會想，沙箱嘛，不就是個容器、Docker 一包就好了嗎？這一期最核心的一個共識，剛好就是要打掉這個想法。到二零二六年，從業者已經有了相當一致的看法：跑 AI 生成的、沒人看過的程式碼，光靠 Docker 這種共享核心的容器，是不夠的。為什麼？因為這些威脅不是假想出來嚇人的，是真的有人被打穿了。我先點幾個名字，後面幾集會慢慢展開。一個叫 EchoLeak，它是第一個在正式的 AI 系統裡，不需要你點任何東西，就能把資料偷走的提示注入攻擊。另一個是 n8n 的漏洞，嚴重程度拿到滿分，一條鏈打通就能執行任意指令、把所有憑證都解出來。還有一個案例代號叫 Claudy Day，它最有意思，因為它根本沒有逃出沙箱，而是沿著沙箱「合法」允許的那條出口，把資料光明正大地送出去。

那這一期我們打算怎麼聊？我們不打算講「哪一家最好」這種空話，而是把沙箱這層基礎設施，拆成五個你可以分開聽懂的切面，順序就照著一到五。

第一集，我們聊為什麼非要沙箱不可。也就是二零二六年的威脅地景到底長什麼樣，「容器不是沙箱」這個共識又是怎麼形成的。你會聽到一個很重要的觀念，就是與其想著找一個完美的沙箱，不如直接假設「你的 agent 遲早會被接管」，然後讓被接管之後的損失是有上限的。

第二集，我們談隔離原語的取捨。市面上有四種把程式碼關起來的辦法——標準容器、gVisor、Firecracker 這種 microVM，還有 WebAssembly。它們之間其實是一個三角拉扯：啟動速度、隔離強度、還有效能開銷，你最多只能挑兩個，第三個一定要讓步。

第三集，我們盤點託管沙箱市場。就是 E2B、Modal、Daytona、Vercel、Cloudflare 這五家，把它們的定價、冷啟動、快照功能全攤開來比，告訴你什麼情況該選誰。你會發現它們的計費方式差到不能直接相減。

第四集，我們把鏡頭拉回你自己的筆電。看 Claude Code、OpenAI 的 Codex，還有一個叫 nono 的工具，怎麼用作業系統內建的機制，把跑在你機器上的 agent 給關住。這裡有個很哲學的對比：你那道沙箱，到底是一面拆不掉的牆，還是一道隨時能被你一句「好、同意」就打開的閘門。

第五集，也是收尾，我們談營運跟架構。把前面所有東西疊起來，當你真的要為產品經營這一層沙箱，快照、分叉、還有四道防禦邊界，到底該怎麼設計。

所以這一期的主軸就是這句話：跑模型已經不難了，難的是安全地跑「模型寫出來的程式碼」。聽完這五集，希望你下次面對「我的 agent 該跑在什麼沙箱裡」這個問題，能給出一個有依據的答案，而不是憑感覺挑一個 logo。那我們就從第一集，威脅地景開始聊起。
