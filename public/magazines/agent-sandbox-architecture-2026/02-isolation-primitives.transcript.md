上一集我們講了為什麼非要沙箱不可,結論是,你得假設 agent 一定會被接管,所以需要一道牆。那這一集,我們就來看砌這道牆有哪幾種材料。也就是隔離原語的取捨。

先把上一集的結論再往前推一步。要安全地跑「模型剛剛寫出來的程式碼」,第一個要推翻的假設,就是「容器等於沙箱」。容器是用 namespace 跟 cgroup,把資源藏起來、限制起來,但容器裡的程式,還是直接對「同一顆主機核心」發系統呼叫。系統呼叫你可以想成是程式向作業系統請求服務的入口,像開檔、連網這些。問題是,核心裡只要有一個邏輯漏洞,就是一次逃逸。二零二五年底就連續爆了三個這類漏洞,讓攻擊者能對主機檔案任意寫入、甚至完整逃出來。對 AI agent 來說,產生並執行沒人審查的程式碼是家常便飯,所以這個風險的暴露面,遠遠高於傳統應用。

於是業界在二零二六年,大致收斂出四條隔離路線。我建議你把它們想成是同一個三角形上的四個角。這個三角的三個頂點是:冷啟動的速度、隔離的強度、還有效能的開銷。重點來了——你最多只能挑兩個,第三個一定要讓步。沒有任何一種方案能同時做到又快、又強、又輕。我們一個一個來看。

第一種,標準容器,就是 runc 那種。它的隔離邊界,就是 Linux 核心本身。好處是幾乎沒有開銷、啟動只要毫秒、什麼 Linux 程式都能跑。代價就是,它的攻擊面等於整顆核心的系統呼叫表,好幾百個進入點。對你自己人寫的程式,這完全夠用;對模型寫的程式,這個共享核心就是一個單點故障。

第二種是 gVisor,它的做法很巧妙。它在中間插了一層「跑在使用者空間的假核心」,這層東西叫 Sentry。容器裡的程式發出的每一個系統呼叫,都先被 Sentry 攔下來,Sentry 自己用 Go 重新實作了一遍核心的行為,只用一個很小的白名單,去對真正的主機核心發呼叫。所以你的程式以為它在跟核心講話,其實它在跟 Sentry 講話,真正的核心攻擊面被大幅縮小了。代價是什麼?是每一次系統呼叫都多一道攔截跟模擬。所以那種系統呼叫很密集、或者網路、I/O 很重的工作,會明顯變慢,大概慢個一到三成;但純運算的工作受影響就小。

第三種,也是這幾年最關鍵的轉折,是 Firecracker 這種 microVM。它乾脆把賭注押在硬體虛擬化上。它是 AWS 用 Rust 寫的一個很輕的虛擬機監控器,透過核心的 KVM 功能,給每一個 microVM 一顆「專屬的」核心。注意,是專屬的,guest 跟 host 不再共用核心了。所以你要逃逸,得先攻破底下的 hypervisor,這個門檻比攻破一個共享核心的邏輯漏洞高太多了。過去大家反對 microVM 只有一個理由:慢,冷啟動要一兩百毫秒。但二零二六年真正的轉折是「快照還原」這個功能。它把一台跑到一半的 microVM,整個記憶體跟裝置狀態存成檔,要用的時候再還原回來,而且還原可以做到三到八毫秒這個級距。這句話的意思是,「為每一個請求開一台專屬虛擬機」這件事,第一次變得實際了。它直接把「強隔離一定慢」這個老論點,打掉了一半。

不過快照不是免費的午餐,它把問題從「速度」搬到了「正確性」。Firecracker 官方自己警告:如果你重複從「同一份快照」還原,會產生危險的重複——因為快照裡可能存了亂數種子、識別碼、甚至快取在記憶體裡的加密 token,你以為在無害地複製,其實是把一份本該唯一的密碼學狀態,散到了好多台虛擬機上。它們用一個叫 VMGenID 的機制,讓 guest 能偵測到自己是被還原出來的、進而重新洗牌亂數,但應用層的去重,還是得你自己負責。這個陷阱第五集我們還會再碰到。

第四種路線最特別,它乾脆繞過整個 Linux 核心,就是 WebAssembly,還有同一個哲學的 V8 isolate。WASM 模組預設是什麼都碰不到的,它摸不到檔案系統、摸不到網路,除非 host 在啟動它的時候,顯式地把某個能力遞給它。這個就叫「能力導向安全」,預設一律拒絕,你給什麼它才能做什麼。V8 isolate 是同樣哲學的引擎級實作,啟動不到五毫秒、每個只佔不到一 MB 記憶體,密度高到不行,在邊緣運算跟 FaaS 已經很成熟了。

但這裡有個常被忽略的逆轉,關於隔離強度。V8 isolate 是「同一個主機程式裡」的多租戶,也就是說,一次逃逸會同時波及這個程式裡所有的租戶;而 Firecracker 的逃逸得攻破 hypervisor。所以二零二六年,不少當初純為了效能選 V8 的平台,開始重新檢視這個選擇值不值得。另外 WASM 的適配也比較硬:它只能跑 JS 跟 WASM,沒有檔案系統、不能跑任意 Linux 程式;而且 WASM 的系統介面 WASI,早期版本連網路都沒有,後來才補上,但多執行緒到現在還是個缺口。所以結論是,WASM 跟 isolate 在邊緣函式很成熟了,但要把那種「會開執行緒、會讀寫檔、會裝任意套件」的真實 agent 工作整碗端進去,還力有未逮。

所以這一集你可以記住一句話,就是把那個三角攤開來問自己:你最在意的,到底是啟動延遲、是能跑什麼東西、還是逃逸之後的爆炸半徑?如果你要跑模型生成的任意 Linux 程式碼、又完全不信任它,那 Firecracker 或它上面的 Kata,是二零二六年多租戶不可信程式碼的防守基線;如果你的工作受控在 JS、WASM,又追求極致的密度跟速度,那就走 isolate;如果你只想在攻擊面跟部署簡便之間找平衡,gVisor 是個務實的中間點;標準容器,就留給你信得過的程式碼。記住,這個三角消滅不了,只能被選擇。那既然原語講完了,下一集我們就來看,市面上那五家託管廠商,各自選了哪個角,又怎麼把它包成生意賣給你。
