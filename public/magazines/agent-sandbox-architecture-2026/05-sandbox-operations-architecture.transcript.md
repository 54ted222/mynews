這是這一輯的最後一集。前面四集,我們講完了為什麼要沙箱、拿什麼隔離、向誰租、還有在筆電上怎麼跑。但這一集我想聊的,是當你真的把沙箱推上產線之後,你會發現它不是一個一次性開起來、用完就丟的容器,而是一層需要被「經營」的基礎設施。它有狀態要管、有生命週期要排程、有好幾個方向的逃逸要堵。我們就站在「我要為產品建一層沙箱」的角度,來談這層東西該怎麼設計。

我先給你兩個核心的心智模型,後面所有東西都掛在這兩個上面。第一,沙箱的狀態——記憶體、檔案系統、執行中的程式——這些是「資產」,而快照、分叉、暫停恢復,是你操作這份資產的工具,用好了能把成本砍掉一個量級。第二,沙箱的對外介面,是「威脅面」,而二零二六年的共識是:別指望 prompt 能擋住資料外洩,唯一可靠的防線,在你自己控制的基礎設施層。

我們先聊第一個,把狀態當資產。agent 工作流跟傳統的請求回應,最大的不同是,它狀態很重、又很愛試錯。一個 coding agent 可能花三十秒裝完依賴、載入一份大型程式庫、跑起語言伺服器,這份「熱」起來的記憶體狀態,如果每次都從零重建,光環境準備就吃掉大半時間。快照就是用來保住這份投資的。

這裡的關鍵分水嶺是:你的快照到底存了什麼?最弱的一檔,只存檔案系統跟套件,恢復之後你拿回的是檔案,但記憶體裡那份載好的模型、解析完的資料集,全部蒸發了,得重跑。較強的一檔,連完整的記憶體跟執行中的程式一起存,像 E2B 那樣,恢復後一模一樣回到暫停前。對那種累積了昂貴記憶體狀態的工作,這個差別不是優化,是質變。而更底層的 microVM,像 Firecracker,把這件事做到毫秒級,還原大概三到八毫秒。它把「強隔離一定慢」這個老論點,徹底打掉了。

但快照真正的威力,是「分叉」。分叉就是複製一份完整的執行狀態,從同一個熱起點,分岔出好多條平行的分支。為什麼 agent 特別需要這個?因為 agent 的探索,本質上是樹狀的。你可以想成下棋的時候,從同一個盤面出發,試好幾種走法,評估完再決定哪一步留下來。各種 agent 推理框架——像思維樹、像蒙地卡羅樹搜尋——全都需要「從同一個中間狀態出發,試很多條路,挑最好的留下」。沒有分叉,你只能對每一條分支,重跑一次完整的環境設定;有了分叉,分岔的成本就趨近於一次記憶體複製而已。

不過,分叉有一個常被忽略的安全陷阱,這個我們第二集稍微提過,這裡得講透。Firecracker 官方明確警告:同一份快照被多次恢復,會讓 guest 作業系統的亂數種子、唯一識別碼、甚至「快取在記憶體裡的密碼學 token」,全部被複製。你以為你在做無害的平行複製,實際上,你把一份本該唯一的密碼學狀態,散播到了好多台虛擬機上。有個叫 VMGenID 的機制能讓系統偵測到自己是被恢復的、進而重新洗亂數,但它救不了那些已經被快取住的種子跟 token。所以分叉不能無腦用:凡是快照裡可能含有有效憑證的場景,恢復之後都得有一套去重的機制——重新注入機密、重新洗亂數、輪換 token——否則你的狀態管理,會反過來變成攻擊面。

好,聊完資產,我們聊威脅面。跑模型寫的程式碼,逃逸不是「會不會」的問題,而是「往哪裡逃」的問題。二零二六年的緩解共識,微軟跟 NVIDIA 的指引在這點上是收斂的,就是把防禦切成四道一定要設的邊界。你可以把它們想成四個方向。

第一道,網路出口,這是對外的線,預設拒絕,只開白名單。第二道,檔案系統邊界,把工作目錄釘死在沙箱裡,主機的敏感路徑——憑證、其他人的資料、build 快取——通通不要掛進來,能唯讀就唯讀。第三道,機密的範圍控管,這是個常被做錯的地方:很多人直接把整份主機的環境變數倒進沙箱,等於把所有憑證交給一段不可信的程式碼。正確做法是「按任務注入」——沙箱只在執行當下,拿到這個任務真正需要的那一把鑰匙,任務結束就撤銷。第四道,設定檔保護,要堵的是「改規則來解除限制」這條路:封掉 agent 對自己設定的修改,不管使用者核可到什麼層級,因為一旦它能改自己的設定,前面三道邊界都可能被它自己拆掉。

這裡有個架構層級的取捨,我覺得是整套設計的核心,一定要講。這四道,你在「哪一層」去強制?微軟那套工具,是在應用的中介層做政策強制,但這代表強制的邏輯本身,跟不可信的程式碼跑在同一個信任域裡。所以它的生產建議是:把每個 agent 跑在獨立的容器裡,拿到作業系統級的隔離,再把中介層的政策疊上去。這給我們一個明確的原則:政策層好用、好稽核、好除錯,但它不是隔離邊界。真正的邊界,必須落在作業系統、hypervisor、網路這些「agent 沒辦法用 prompt 去影響」的層級。換句話說,把「可以被說服的東西」——也就是 LLM——跟「不能被說服的東西」——也就是封包路由、檔案掛載——徹底分開,這是整套架構的靈魂。

四道邊界裡,我想特別講網路出口,因為它是最後一道線,也最容易被做成空殼。前面三道做好,攻擊者拿到的東西就被困在沙箱裡了;但只要 agent 還能對外發任何一個封包,被偷的資料就有出口。所以出口控制要堵的,就是最後這一跳。設計原則只有一句:預設封鎖所有對外連線,只白名單真正需要的,而且強制點要「fail closed」——在代理層攔,非白名單的直接連不出去,而不是「先連了再記 log」。有兩類目的地一定要明確封死:一個是雲端的中繼資料端點,那個固定位址,連到它就能拿到主機的憑證去做橫向移動;另一個是內網段。

但這也是反方觀點最該被聽見的地方:白名單寫下來,不等於它生效。實務上有好幾種繞過手法,我舉兩個你感受一下。一種是 DNS 重綁定:它在你「檢查的時候」解析到一個白名單的合法 IP,騙過檢查,但在「真正連線的時候」,已經把同一個網域重新指向了內網。要徹底擋住,你得在傳輸層把已經解析出來的那個 IP 給釘死。另一種是解析器差異,利用不同元件對同一個字串的理解不一致來鑽縫。這些全都得當成邊界的一部分,逐項測過。所以我的建議是:把出口控制當成跟程式碼一樣、要寫測試的東西。對每一條「允許」規則,都寫一組「應該被擋下」的負向測試丟進 CI;並且在代理側留下可稽核的紀錄,讓「誰連了哪裡」變成從沙箱外部就能驗證的證據,而不是去信任沙箱裡面那份 log。

最後講講延遲、成本跟安全的取捨,因為你把這些東西全疊起來,一定會撞到三角:每多一層隔離跟檢查,就多一點延遲跟成本。誠實面對它,比假裝沒有開銷重要。延遲面有個好消息:強隔離不再等於慢了,microVM 的還原已經進到幾毫秒級,所以「為了安全要忍受很差的啟動體驗」這個論點,二零二六年大致站不住了,前提是你善用快照跟分叉把冷啟動攤掉。成本面,最值得投資的是狀態管理:暫停恢復讓 agent 在等審核的時候不燒算力,分叉讓平行探索的成本,從「N 次完整環境」降到「一份基底加 N 份差異」。

所以,把整輯收束一下。我們從第一集的威脅地景,知道了得假設 agent 一定會被接管;第二集看了四種隔離原語的三角取捨;第三集比了五家託管市場;第四集回到你的筆電,辨認那道牆到底是牆還是閘門;到這一集,我們把它經營成一層真正的基礎設施。沙箱會是 agent 堆疊裡最貴的一層,這點沒變。但這一輯想告訴你的是:只要你把狀態原語用好、把四道邊界落在對的層級,這層投資買到的,是一件二零二六年已經不再是「有了更好」、而是「沒有不行」的事——那就是,能放心地讓模型寫出來的程式碼,真的上線。
