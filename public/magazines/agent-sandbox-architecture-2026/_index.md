2026 年中，AI agent 的競爭軸線悄悄換了一條。模型本身的差距在縮小，真正卡住產品化的是另一個問題：**當 agent 會自己寫程式、自己執行、自己連網，你要把它關在哪裡？** 一份產業分析直接把沙箱稱為「agent 堆疊裡最被爭奪的一塊地」——因為它同時牽動安全邊界、冷啟動延遲與每月帳單，而這三者很難同時要到。

威脅不是假想。2025 到 2026 年間，EchoLeak（CVE-2025-32711，CVSS 9.3）成為第一個被武器化、能在正式系統裡零點擊外洩資料的 prompt injection；n8n 的 CVE-2026-25049 拿到滿分 10.0，一條鏈打穿就能執行任意指令、解密所有憑證。到了 2026 年 2 月，從業者的共識已經很白：**跑 AI 生成的程式碼，光靠 Docker/runc 不夠**。於是 microVM、gVisor、WebAssembly 各自卡位，E2B、Modal、Daytona、Vercel、Cloudflare 把沙箱包成託管服務在賣，Claude Code 與 OpenAI Codex 則把沙箱直接做進你筆電上的 CLI。

這一期我們不談「哪家最好」這種空話，而是把這層基礎設施拆成五個能各自讀透的切面：第一篇講**為什麼非沙箱不可**——2026 的威脅地景與「容器不是沙箱」的共識怎麼形成；第二篇講**隔離原語的取捨**，從容器、gVisor、Firecracker microVM 到 WASM，冷啟動與隔離強度為何此消彼長；第三篇盤點**託管沙箱市場**，把定價、快照、分叉攤開來比；第四篇回到**你自己的機器**，看 Claude Code、Codex、nono 怎麼用 OS 原語把 agent 關起來；第五篇談**營運與架構**，快照、分叉與分層防禦怎麼設計成真正可上線的一層。讀完這期，你應該能對「我的 agent 該跑在什麼沙箱裡」給出有依據的答案，而不是憑感覺選一個 logo。
