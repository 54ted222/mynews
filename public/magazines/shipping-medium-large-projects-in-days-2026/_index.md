這一期談的不是「AI 會不會寫程式」，而是更現實的問題：**你能不能在 3 到 7 天內，交付一個可用、可驗證、可維護的中大型專案切片**。很多人看到社群貼文，會以為關鍵在於某一段超長提示詞；但把官方文件和失敗案例放在一起看，答案其實非常一致：快，來自流程；穩，來自護欄。

Anthropic 在 Claude Code 文件裡，把 `CLAUDE.md`、Plan Mode、hooks、subagents、與工作流程拆分，全部視為同一套工程化方法，而不是附加功能。這代表它的設計假設並不是「讓你一句話做完整系統」，而是「讓你把需求壓成規格，再用可驗證的小步驟前進」。換句話說，AI 只是執行器，真正決定成功率的是你有沒有把規則外部化。

這一點在外部案例也重複出現：有人把冗長需求先壓縮成結構化規格，才在極短時間內交出可用產品；也有人在不到一週內做完 full-stack MVP，但一上真實使用者就暴露效能、查詢與行動版問題；還有把生產環境基礎設施操作交給 agent，結果觸發嚴重資料事故。它們不是互相矛盾，而是剛好拼出同一張圖：**做出功能很快，做出可靠系統很慢；中間的差距要靠流程補**。

因此，本期用四篇文章把「幾天完成中大型專案」拆成一條可複製的路徑：先決定什麼叫完成，再安排 7 天節奏；接著把需求轉成 SPEC 與 Plan Mode 的只讀分析；再把測試、hooks、review、CI 做成機械化關卡；最後用真實事故回推什麼必須由人類保留最終決策權。

如果你只記得一句話，請記這句：**把 Claude Code 當會寫程式的同事，不是當會負責任的主管。** 速度可以委派，但責任、資料風險與上線決策不能外包。
